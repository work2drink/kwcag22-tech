<Aside title="검사 항목">
인증 과정은 인지 기능 테스트에만 의존해서는 안 된다.
</Aside>

## 목적
이 검사 항목의 목적은 사용자가 계정 로그인 등의 인증 과정을 수행할 때, 인지 부담이 큰 요소(예: [CAPTCHA](../../glossary/captcha), 복잡한 퍼즐, 임의 숫자 기억 등)에 의존하지 않고, 접근 가능하고 사용하기 쉬우며 안전한 인증 방법을 제공하는 것이다. 이를 통해 인지 능력이나 기억력에 제한이 있는 사용자도 부담 없이 인증을 완료할 수 있도록 돕는다.

##	배경 및 필요성
가장 널리 사용되는 아이디와 비밀번호 인증 방식은 사용자가 정보를 기억해야 하므로, 인지 능력이나 기억력이 제한된 사용자에게 어려움을 줄 수 있다.

또한, [일회성 확인 코드(OTP)](../../glossary/otp)를 입력하거나, [CAPTCHA](../../glossary/captcha)나 퍼즐, 왜곡된 문자 입력 등 인지 기능 테스트 방식은 난독증이나 인지 장애가 있는 사용자에게 추가적인 부담을 주어 인증을 어렵게 만든다.

이를 위해 사용자가 인지 기능 테스트에 의존하지 않고 여러 대체 인증 수단 중에서 선택할 수 있도록 하는 것이 중요하다.

##	기본 규칙
인증 과정에서는 보안과 접근성 사이의 균형을 유지하면서, 사용자의 인지 또는 학습 능력에 의존하지 않는 다양한 인증 방식 대안을 제공해야 한다.

사용자 인증 과정에서 인지 기능 테스트([CAPTCHA](../../glossary/captcha), 퍼즐, 복잡한 문자 입력 등)에 의존하는 경우, 반드시 인지 부담이 적은 대체 인증 방법을 함께 제공해야 한다. 이를 위해 다음과 같은 방안을 고려할 수 있다.
- 브라우저가 아이디/비밀번호를 저장할 수 있도록 허용하여, 사용자가 매번 정보를 기억할 필요 없도록 한다.
- 비밀번호 관리 프로그램이 자동으로 정보를 채워 넣을 수 있도록 허용한다.
- 비밀번호, 이메일로 전달받은 [OTP](../../glossary/otp) 등 사용자가 해당 내용을 복사하여 붙여 넣을 수 있도록 허용한다.
- 생체 인증(예: 지문, 얼굴 인식) 또는 물리적 인증(예: USB 토큰, 휴대폰 인증 등)을 제공한다.
- 이메일이나 SMS를 통해 로그인 가능한 인증 링크를 제공한다.
- 공개 인증([oAuth](../../glossary/oauth))과 같은 타사 인증 방식을 통해 사용자가 로그인할 수 있도록 한다.

## 적용 사례
### 1)	자동 완성 및 복사/붙여넣기 허용
비밀번호 관리 프로그램을 사용하여 비밀번호 입력란을 자동으로 채울 수 있도록 허용하여, 사용자가 정보를 직접 기억하고 입력하는 부담을 줄인다. 또한, 추가 인증 과정에서 이메일로 OTP 코드를 발송하는 경우, 사용자가 단순히 복사하여 붙여 넣을 수 있도록 기능을 허용함으로써 인지 능력에 대한 의존도를 낮춘다.

다음은 메일로 전송된 인증 코드를 복사하여 인증 코드 입력창에 붙여 넣을 수 있도록 허용한 사례이다.

### 2)	공개 인증(oAuth)을 통한 인증 제공
사용자에게 익숙한 소셜 미디어나 이메일 계정을 통해 인증할 수 있도록 oAuth와 같은 공개 인증 방식을 제공한다.

![이미지. 타사 계정을 통한 로그인 제공 사례 (출처: 마켓컬리)](/images/image249.png)

### 1)	사용자 기기를 활용한 인증 제공
기기에 저장된 [패스키](../../glossary/passkey)를 사용하여 복잡한 비밀번호 입력 없이 인증을 완료할 수 있도록 제공한다. 이는, 스마트폰에 저장된 패스키를 사용하여 복잡한 비밀번호 입력 없이 인증을 완료할 수 있게 한다.

단, QR 코드 스캔 등 시각적 입력이 필요한 경우에는 시각을 활용하기 어려운 사용자를 위한 대체 방안을 제공해야 한다.

![이미지. 기기 간 패스키를 통한 대체 인증 제공 예시](/images/image250.png)

### 4)	이메일 링크 인증
이메일 링크 인증은 사용자가 인지 능력을 과도하게 사용하지 않고도 계정에 접근할 수 있도록 돕는 인증 방식이다. 

기본적으로 사용자 이름과 비밀번호를 통한 로그인 방식을 제공하되, 비밀번호를 잊었을 경우 이메일을 통해 로그인 가능한 별도의 인증 링크를 제공한다. 사용자가 자신의 이메일 주소를 입력하면, 사이트는 해당 이메일로 인증 링크를 전송하며, 사용자는 이메일에 포함된 링크를 클릭하기만 하면 복잡한 절차 없이 바로 웹 사이트에 로그인할 수 있다.

이 방법을 통해 사용자는 복잡한 비밀번호를 기억하거나 퍼즐을 푸는 등의 인지 부담을 줄일 수 있으며, 별도의 인지 기능 테스트 없이도 안전하고 편리하게 인증을 완료할 수 있다. 이러한 접근 방식은 다양한 사용자에게 편의를 제공하며, 접근 가능한 인증을 구현하는 대표적인 사례가 된다.

![이미지. 이메일로 인증할 수 있는 사례 (출처: Medium)](/images/image251.png)

![이미지. 접속할 수 있는 인증 링크가 담긴 메일 사례 (출처: Medium)](/images/image252.png)

## 개선 필요 사례
### 1)	아이디, 비밀번호 입력으로만 인증
사용자가 매번 아이디와 비밀번호를 기억하여 입력해야 하므로, 인지 능력이 제한된 경우 로그인 과정에서 어려움을 겪을 수 있다.

![이미지. 아이디, 비밀번호 외 다른 인증 수단이 제공되지 않은 로그인 예시](/images/image256.png)

#### 개선 방법
브라우저 저장 기능 등을 활용하고, 대체 인증 방법을 함께 제공한다.

### 2)	인지 기능 테스트에만 의존한 인증 방식
다음은 게시글 자동 등록 방지를 위해 인지 기능 테스트(CAPTCHA)로 사람임을 인증해야 하는 사례이다.

왜곡된 문자 그림이나 빠르게 읽어주는 음성을 듣고 기억하여 입력해야 하므로 난독증 등 일부 사용자에게 매우 부담스러울 수 있다.

![이미지. 음성을 듣고 기억하여 입력하는 자동등록방지 사례](/images/image257.png)

#### 개선 방법
사용자가 인지 부담이 적은 인증 방식을 제공한다. 예를 들어 '나는 로봇이 아닙니다'와 같은 체크박스 기반 ReCAPTCHA v3 등 대체 인증 방법이 있다.

### 3)	캡챠와 인지 테스트를 같이 요구하는 방식
이미지를 다운로드하기 위해서 캡챠 문자를 입력하고, 지시사항에 맞춰 캡챠 내 문자를 차례대로 선택해야 하는 방식은 시각 장애인, 난독증 등 일부 사용자는 해당 테스트를 수행하기 어렵다.

![이미지. 캡챠 내 문자를 요구한 순서대로 선택해야 하는 사례](/images/image258.png)

#### 개선 방법
시각 장애인을 포함한 모든 사용자가 접근할 수 있도록, 인지 부담이 적은 대체 인증 방법을 제공한다.

### 4)	입력란에 붙여넣기 기능 차단
다음은 스크립트를 사용하여 비밀번호 입력란에 붙여넣기 기능을 차단한 코드가 크롬 Lighthouse 접근성 검사에서 검출된 모습을 보여준다.

비밀번호 입력란에 `onpaste="event.preventDefault();"`와 같은 기능을 사용하면, 사용자는 브라우저의 자동 저장 기능이나 비밀번호 관리 프로그램 등을 활용할 수 없어 인증 과정이 불편해질 수 있다.

또한 복잡한 비밀번호 또는 인증 번호 등을 직접 입력해야 하므로 인지적 부담이 증가하고 입력 과정에서 오류가 발생할 가능성이 커진다.

![이미지. 구글 크롬의 Lighthouse로 접근성 검사 시, 붙여넣기 차단 코드 검출 사례 (출처: chrome for developers)](/images/image259.png)

#### 개선 방법
자바스크립트 등으로 인증 입력란에서 붙여넣기 기능을 차단하지 않는다.
