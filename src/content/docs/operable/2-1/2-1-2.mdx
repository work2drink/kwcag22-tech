<Aside title="검사 항목">
키보드에 의한 초점은 논리적으로 이동해야 하며, 시각적으로 구별할 수 있어야 한다.
</Aside>

## 목적
키보드 초점 이동과 표시는 사용자가 키보드만으로 웹 페이지의 모든 기능을 예측 가능한 논리적 순서로 탐색할 수 있도록 보장하며, 현재 초점이 위치한 요소를 시각적으로 명확하게 표시하여 콘텐츠와의 원활한 상호작용을 돕는다.

##	배경 및 필요성
키보드만 사용하는 사용자는 마우스 없이 콘텐츠를 탐색해야 하므로, 논리적이고 예측 가능한 초점 이동 순서가 필수적이다. 참고로, 화면낭독프그램 사용자도 동일하게 키보드를 사용하여 웹을 탐색한다.

예를 들어, 잘못된 tabindex 설정이나 스크립트 오류로 인해 초점이 올바르지 않은 순서로 이동하면, 사용자는 의도한 버튼이나 링크 대신 다른 요소에 초점이 표시되며, 원하는 요소에 도달할 때까지 불필요한 탐색을 반복해야 한다.

초점이 시각적으로 명확하게 표시되지 않으면, 사용자는 현재 어느 요소(예: 버튼, 링크, 입력란)에 초점이 있는지 파악하기 어려워 원하는 메뉴로 이동하는 것조차 어려워진다.

또한, 비활성화된 ‘완료’ 버튼이나 숨겨진 요소에 초점이 머무르면, 키보드 사용자와 화면낭독프로그램 같은 보조 기기 사용자는 해당 요소가 사용 가능한 상태라고 오해하여 잘못된 조작을 시도할 수 있다.

이러한 문제를 예방하기 위해, 초점 이동이 일관되고 논리적인 순서로 이루어지도록 설계하며, 숨겨진 요소나 비활성 요소에는 초점이 이동하지 않도록 해야 한다.

##	기본 규칙
다음 사항들을 충족해야 한다.
- 키보드로 초점을 받은 요소(버튼, 링크, 입력란 등)는 테두리, 배경색 변화 등 명확한 시각적 피드백을 제공해야 하며, 초점 표시가 충분히 눈에 띄어야 한다.
- 키보드로 콘텐츠를 탐색할 때, 초점은 사용자가 예측할 수 있는 논리적인 순서로 이동해야 한다. 일반적으로 콘텐츠는 좌측에서 우측, 상단에서 하단 순으로 탐색된다.
- tabindex="1" 이상의 값을 사용하면 초점 이동 순서가 강제로 변경되어 마크업 순서와 불일치하게 된다. 따라서, 특별한 경우가 아니면 tabindex 사용을 지양해야 한다.
- 일부 UI 요소는 tab 키 외에 다른 탐색 방법이 필요할 수 있다. 예를 들어, 동일한 그룹 내 라디오 버튼은 좌/우 방향키로, 날짜 선택(datepicker)UI에서는 날짜를 탐색할 때 상/하/좌/우 방향키로 초점을 이동하는 방식이 사용된다.
- 초점이 비활성 또는 숨겨진 요소로 이동하지 않아야 한다.
- 조작 중 특정 요소에 초점이 고정되어 다른 콘텐츠로 이동할 수 없는 경우가 발생하지 않도록 해야 한다.
- 모달 팝업의 경우, 초점은 팝업 내부에 머물러야 하며, 팝업 종료 후에는 논리적으로 적절한 위치로 복귀하도록 설계한다.

</dl>

## 적용 사례

### 1)	논리적 초점 이동
로그인 화면에서 첫 번째 초점이 이동 가능한 아이디 입력란부터 순차적으로 초점이 이동하는 사례이다.

초점 이동 순서는 HTML 마크업 순서를 따르므로, 마크업 시 논리적인 순서를 고려해야 한다. 이와 같이 입력란 → 비밀번호 입력란 → 로그인 버튼 순으로 초점 이동 순서가 사용자 예상과 일치하면 화면낭독프로그램 사용자와 키보드 사용자는 콘텐츠를 쉽게 탐색하고 필요한 작업을 수행할 수 있다.

<figure>
	![](/images/image125.png)
	<figcaption>이미지. 로그인 시 필요한 입력 흐름을 따라 초점 이동 순서가 구현된 예시</figcaption>
</figure>

```html
<form>
	<label for="username">아이디</label>
	<input type="text" id="username">
	<label for="password">비밀번호</label>
	<input type="password" id="password">
	<button type="submit">로그인</button>
	<button type="button">회원가입하기</button>
	<button type="button">아이디, 비밀번호 찾기</button>
</form>
```
### 2)	모달 팝업에서의 초점 이동
모달 팝업은 화면의 다른 콘텐츠와 독립된 콘텐츠 영역이므로, 초점은 모달 내부에서만 이동해야 하며 외부 콘텐츠 영역으로 이동하지 않도록 설계해야 한다.

모달 창이 열릴 때 초점은 모달 팝업 내부의 첫 번째 상호작용 요소로 이동하고, Tab 키를 누르면 초점이 모달 내부에서만 순환해야 한다. 모달이 닫힌 후에는 초점이 논리적으로 적절한 위치(예: 팝업을 열었던 버튼)로 복귀해야 한다.

다음 사례와 같이 모달 팝업이 활성화되면 순방향 초점 이동은 1 → 2 → 3 → 4 → 5 → 1, 역방향은 1 → 5 → 4 → 3 → 2 → 1의 순환 구조로 이동해야 한다.

### 3)	보이는 요소에만 초점 접근
키보드 초점은 화면에 실제로 표시되는 상호작용 요소에만 적용되어야 하며, 숨겨진 콘텐츠로 이동하지 않아야 한다. 예를 들어, 캐러셀 배너 영역에서 여러 개의 배너가 하나씩 순차적으로 노출될 경우, 현재 보이지 않는 배너에는 초점이 이동하지 않아야 한다. 

숨겨진 콘텐츠에 초점이 이동하면, 사용자는 이를 인식하지 못하거나 예상치 못한 동작을 경험할 수 있다. 다만, ‘본문 바로가기’ 링크처럼 기본적으로 화면에 보이지 않지만, 초점을 얻었을 때 노출되는 경우는 사용자가 현재 초점의 위치를 파악할 수 있으므로 예외로 한다.

## 개선 필요 사례

### 1)	키보드 초점 미노출
페이지 내 모든 상호작용 요소에 키보드 초점이 시각적으로 표시되지 않는 사례이다.

디자인적 이유로 초점 표시를 제거할 경우, 키보드 사용자는 현재 어느 요소(버튼, 링크, 입력란)에 초점이 있는지 시각적으로 확인할 수 없다. 이로 인해 사용자가 원하는 작업을 완료하지 못하거나, 초점 위치를 예측하기 어려워 반복적으로 키보드를 눌러야 하므로 탐색 시간이 길어지고 피로도가 증가할 수 있다.

<figure>
	![](/images/image128.png)
	<figcaption>이미지. CSS에서 초점 표시를 제거한 사례</figcaption>
</figure>

```css
a:focus, button:focus, input:focus, label:focus, select:focus {
	outline: none;
}
```

### 개선 방법
초점 표시를 제거하거나 비활성화하지 않고, 기본 초점 스타일을 유지한다. 또는 사용자 인터페이스와 조화를 이루는 초점 스타일을 변형하여 시각적으로 명확하게 표시한다.

### 2)	논리적이지 않은 초점 이동
펼쳐진 하위 메뉴에서 키보드 초점이 논리적인 순서로 이동하지 않는 사례이다.

'알바' 메뉴의 하위 메뉴가 펼쳐졌을 때, `Tab` 키를 누르면 예상과 다르게 '알바' 메뉴 → 서브 메뉴 마지막 항목 ‘신뢰와 안전’ → '동네 업체'로 초점이 이동하여, 사용자가 일부 서브 메뉴 항목을 인식하지 못하는 경우가 발생한다.

<figure>
	![](/images/image129.png)
	<figcaption>이미지. 메뉴 확장 후 Tab 키에 의한 논리적이지 않은 초점 이동 순서 사례</figcaption>
</figure>

#### 개선 방법
서브 메뉴가 확장되었을 때, 서브 메뉴의 첫 번째 항목부터 순차적으로 초점이 이동하도록 구현한다. 그리고 서브 메뉴 전체를 탐색한 후, 다음 주요 메뉴로 이동하도록 설정한다.

### 3)	숨겨진 콘텐츠로 초점 이동
CSS로 숨긴(<span lang="en">hidden</span>) 비활성 상태의 콘텐츠로 초점이 이동하는 사례이다.

사용자는 초점이 현재 화면에 보이는 콘텐츠 내에서만 이동할 것이라 예상하지만, 화면에 표시되지 않는 가려진 배너 목록으로 초점이 이동하면서 현재 위치를 파악하기 어려워진다. 숨겨진 콘텐츠로 초점이 이동하면 사용자는 현재 초점의 위치를 인식하지 못해 예상치 못한 조작 오류가 발생할 수 있다.

초점이 가려진 영역으로 이동하면, 시각적으로 보이는 배너와 초점을 받은 배너가 일치하지 않을 수 있으며, 이로 인해 사용자는 초점이 사라졌다고 오해하거나 탐색이 중단되었다고 느낄 수 있다.

<figure>
	![](/images/image130.png)
	<figcaption>이미지. overflow: hidden 속성에 의해 화면에 노출되지 않는 영역에 초점이 이동한 사례</figcaption>
</figure>

#### 개선 방법
키보드 초점 이동 대상은 화면에 표시된 콘텐츠로 한정하고, 숨겨진 콘텐츠나 비활성 콘텐츠는 초점 이동 대상에서 제외하도록 설정한다.

예를 들어, `display:none`을 사용하면 자동으로 초점 이동 대상에서 제외되지만, `visibility:hidden`이나 `position:absolute`, `z-index` 등으로 화면에서만 가려진 요소는 추가적으로 CSS 또는 ARIA를 활용하여 초점이 이동하지 않도록 설정해야 한다.

### 4)	tabindex 속성 오용으로 인한 논리적이지 않은 초점 이동 순서
로그인 입력 양식에서 `tabindex` 속성에 1 이상의 값이 지정되어, 키보드 초점 이동 순서가 마크업의 논리적 순서와 다르게 설정된 사례이다.

초점 이동 순서가 이메일 주소 입력란 → 비밀번호 입력란 → 상단 로컬 메뉴(고객센터, 마이페이지 등) 순으로 이동한 후, 다시 페이지의 첫 부분으로 이동하여 키보드 사용자가 예상치 못한 순서로 초점이 이동하는 혼란이 발생한다. 또한, 화면낭독프로그램 사용자도 초점의 현재 위치 파악과 콘텐츠 조작이 어려워진다.

<figure>
	![](/images/image131.png)
	<figcaption>이미지. tabindex 값의 사용으로 논리적이지 않은 초점 이동 순서 사례</figcaption>
</figure>

#### 개선 방법
`tabindex` 속성을 제거하여, 마크업 순서에 따라 자연스럽게 초점이 이동하도록 한다.

### 5)	모달 팝업 외부로 초점 이동
페이지 접근 시 모달 팝업이 즉시 활성화되지만, 초점이 팝업 외부의 어둡게(<span lang="en">dimmed</span>) 가려진 콘텐츠 영역으로 이동하는 사례이다.

초점이 팝업 내부로 이동하지 않으면, 화면낭독프로그램 사용자는 팝업 내부의 콘텐츠를 탐색하거나 조작하기 어려워진다. 키보드 사용자도 팝업 내부로 초점을 이동시키기 위해 불필요하게 많은 이동을 해야 한다.

<figure>
	![](/images/image132.png)
	<figcaption>이미지. 모달 팝업 외부로 초점이 이동되는 사례</figcaption>
</figure>

#### 개선 방법
모달이 활성화되면 초점을 팝업 내부의 첫 번째 상호작용 요소로 이동시키고, 팝업이 닫히기 전까지 초점이 외부 콘텐츠로 이동하지 않도록 한다.

팝업 종료 후에는 초점은 원래대로 페이지 처음부터 탐색할 수 있도록 한다.

### 6)	상호작용 요소가 아닌 텍스트에 초점 이동
상호작용 요소가 아닌 텍스트에 키보드 초점이 접근되는 사례이다.

기능이 없는 텍스트 콘텐츠에 `tabindex` 속성이 적용된 후, Javascript로 `tabindex="0"`으로 강제 변환되도록 구현되어 있다. 이로 인해, `tabindex`가 활성화되어 불필요한 키보드 초점 이동이 발생하여 사용자가 더 많은 초점 이동을 해야 하는 상황이 발생한다.

이는 키보드 사용자에게 탐색의 피로도를 증가시키고 작업 흐름을 방해하며, 이를 상호작용 가능한 요소로 오인할 가능성이 높아진다.

<figure>
	![](/images/image133.png)
	<figcaption>이미지. 상호작용 요소가 아닌 텍스트에 초점이 접근되는 사례</figcaption>
</figure>

```html
<div class="sectionEstimate">
	<div class="textBlue">
		<div tabindex="6">
			선착순 <br> 역대급 혜택!
		</div>
		<div tabindex="7">
			예상견적 확인하고, <br> 선착순 무료 혜택 받아가세요!
		</div>
	</div>
	<img src="..." alt="프리미엄 패키지 쿠폰 그림">
</div>
...
```

#### 개선 방법
상호작용 요소가 아닌 콘텐츠에 키보드 초점이 접근하지 않도록 tabindex 속성과 관련 Javascript를 제거한다.

<Card title="tabindex 속성에 대해" icon="open-book">
`tabindex` 속성은 HTML 요소가 Tab 키 탐색 순서를 제어하거나 키보드 초점을 받을 수 있도록 설정하는 역할을 한다. 이를 통해 개발자는 다음과 같은 동작을 지정할 수 있다.

- `tabindex="0"`
	- 기능: 기본적으로 초점을 받지 않는 요소(`div`, `span` 등)에 초점을 줄 수 있게 하며, HTML 문서 내 작성 순서에 맞춰 자연스럽게 탐색된다.
	- 사용 예: 접근성 위젯, `div` 등 기본 초점이 없는 요소에 초점을 부여할 때
	- 주의 사항: 너무 많이 상호작용 요소가 아닌 요소에 적용하면 사용자는 혼란스러울 수 있다.
- `tabindex="-1"`
	- 기능: 해당 요소를 Tab 키 탐색 순서에서 완전히 제외한다. 다만, Javascript를 통해 프로그래밍적으로 방향키나 다른 키로 조작할 수 있도록 초점을 줄 수 있다.
	- 사용 예: 숨겨진 메뉴, 콤보박스의 선택 목록 등 자동으로만 초점을 이동해야 하는 곳 또는 당장은 불필요한 영역
	- 주의 사항: Tab 키로 직접 탐색할 수 없으므로, Javascript로 초점 이동이 필요할 때만 사용한다.
- `tabindex="1"` 이상의 양수
	- 기능: 문서의 기본 순서를 무시하고, 지정한 숫자가 작은 요소부터 우선적으로 초점 이동한다.
	- 사용 예: 검색포털 사이트에서 검색창 등 특정 영역을 제일 먼저 탐색하도록 강제해야 할 때 등 최소한으로 사용
	- 주의 사항: 문서 흐름을 뒤섞어 유지보수를 어렵게 만들 수 있어, 가급적 사용을 지양한다. 그리고 모든 양수 `tabindex`를 가진 요소가 초점을 받은 후에야 기본 탐색으로 돌아간다. 

즉, `tabindex` 속성을 사용하면 키보드 초점을 받을 수 없는 요소에 초점을 부여하거나 탐색 순서를 제어할 수 있다. 그러나 유지보수와 접근성을 고려하여 기본 흐름을 크게 변경하지  않는 범위에서 활용하는 것이 중요하다.
