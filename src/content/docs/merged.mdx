# 1. 인식의 용이성

인식의 용이성이란 장애나 환경적 제한과 관계없이 사용자가 웹사이트에서 제공하는 콘텐츠를 쉽게 인지할 수 있도록 하는 것이다. 예를 들어, 시각 장애인은 이미지를 볼 수 없으며, 청각 장애인은 오디오를 듣기 어렵다. 또한, 강한 햇빛 아래에서 화면을 보기 어려운 환경이나 시끄러운 장소에서도 콘텐츠를 문제없이 전달받을 수 있어야 한다.

다음과 같은 상황에서 도움이 될 수 있다.

-	야외나 조도가 낮은 실내에서 화면 빛 반사와 밝기 차이로 인해 텍스트를 구분하기 어려울 때, 충분한 명도 대비가 적용된 디자인 덕분에 내용을 놓치지 않을 수 있다.
-	이모지나 아이콘이 많은 게시물에서도, 장식 목적과 의미 전달 목적을 구분하여 대체 텍스트를 처리하면, 화면낭독프로그램이 불필요한 내용을 읽지 않아 사용자의 피로도를 줄일 수 있다.
-	동영상을 시청할 때, 이어폰 없이 조용히 듣기 어려운 환경에서도 자막이나 원고가 제공되면, 소리를 끄고도 콘텐츠를 정확히 이해할 수 있다.
-	차트나 그래프가 포함된 경우, 텍스트 설명이나 데이터 테이블을 함께 제공하면 시각 장애 사용자도 의미 있는 정보를 전달받을 수 있다.

이처럼 인식의 용이성을 높이면, 가시성과 가독성이 개선되어 전반적인 사용자 경험이 향상된다.

## 1.1 대체 텍스트

<Aside title="검사 항목">
텍스트 아닌 콘텐츠는 그 의미나 용도를 인식할 수 있도록 대체 텍스트를 제공해야 한다.
</Aside>

### 목적
대체 텍스트는 이미지, 차트, 다이어그램, 오디오 등 텍스트가 아닌 콘텐츠가 전달하는 의미나 용도를 인식할 수 있도록 제공하는 간결한 텍스트이다. 이를 통해 시각이나 청각 등의 감각에 제한이 있는 사용자도 동등한 정보를 얻을 수 있다.

##	배경 및 필요성
이미지를 볼 수 없는 사용자는 화면낭독프로그램이 읽어주는 대체 텍스트를 통해 내용을 이해할 수 있으며, 소리를 들을 수 없는 사용자는 화면에 표시된 대체 텍스트를 통해 내용을 확인할 수 있다.

대체 텍스트가 제공되면, 사용자 에이전트(브라우저 등)나 보조 기술이 다양한 방식으로 콘텐츠 정보를 사용자에게 제공할 수 있다. 이를 통해 시각, 청각 등 감각적 제약에 관계없이 모든 사용자가 동등한 정보를 얻을 수 있도록 돕는다.
##	기본 규칙
#### 대체 텍스트 작성 시
- 콘텐츠의 의미와 기능을 정확하고 동등하게 전달할 수 있도록 작성한다.
- 불필요한 정보나 장황한 설명을 피하고, 콘텐츠 맥락을 고려하여 간결하고 명확한 표현을 사용한다.
- 이미지 주변 텍스트와 동일한 정보를 중복 제공하지 않는다.
- 이미지 대체 텍스트에 ‘~이미지’, ‘~사진’, ‘~아이콘’ 등의 불필요한 단어를 포함하지 않는다.

#### 의미 있는 배경 이미지의 경우
- 배경 이미지가 중요한 의미를 전달하는 경우, CSS로 숨김 텍스트를 추가하거나 ARIA 속성 등을 활용하여 보조 기술이 콘텐츠의 의미를 인식할 수 있도록 대체 텍스트를 제공한다.

#### 긴 설명이 필요한 경우
- 그래프, 차트 등과 같이 긴 설명이 필요한 콘텐츠는 해당 콘텐츠 근처에 자세한 설명 텍스트를 제공하거나, 페이지 내 설명을 aria-describedby 속성으로 연결하거나, figure 및 figcaption 요소를 활용하여 설명을 제공할 수 있다.
- 또는, 콘텐츠 상단에 별도의 설명 페이지 링크를 제공하여 사용자가 추가 정보를 확인할 수 있도록 하는 방법도 고려할 수 있다.
- longdesc 속성은 HTML5에서 제거되었으므로, DOCTYPE이 HTML5인 경우 사용할 수 없다.

#### 기능을 가진 경우
- 버튼, 링크 등 상호작용 요소에 사용된 이미지는 해당 기능을 직관적으로 이해할 수 있는 단어로 제공한다. 예를 들어, 돋보기 아이콘은 ‘돋보기 아이콘’이 아닌 alt="검색"으로 작성한다.
- 동일 링크 내 이미지와 텍스트가 함께 있는 경우, 중복된 정보 제공을 피하고자 이미지는 alt 속성을 빈 값(alt=””)으로 제공해야 한다.

#### 의미 없는 장식용 이미지인 경우

- 디자인 목적의 장식 이미지는 보조 기술에서 무시되도록 alt 속성을 빈 값으로 제공한다.    (예: `<img src="bg-pattern.jpg" alt="">`)

다음과 같은 경우에는 예외적으로 대체 텍스트를 제공하지 않거나 제한적으로 제공할 수 있다.

- 대체 콘텐츠로써 제공된 경우: 이미 다른 대체 수단(예: 수어 영상, 그래프 하단의 표 또는 텍스트 설명 등)으로 동일한 정보가 제공된 경우, 추가적인 대체 텍스트는 필요하지 않다. 예를 들어, 비디오 콘텐츠에 수어 영상이 포함된 경우 해당 수어 영상에 대한 별도의 대체 텍스트는 제공하지 않아도 된다.
- 생방송 콘텐츠: 생방송 비디오(또는 오디오)처럼 내용을 미리 알 수 없고, 지속적으로 변하는 경우, 해당 콘텐츠의 용도를 간략하게 설명하는 대체 텍스트만 제공한다.
- 시험이나 테스트: 시력검사, 청각검사 등과 같은 시험의 경우, 정답을 그대로 제공하면 안 되므로, 콘텐츠의 간략한 용도만을 설명하는 대체 텍스트를 제공한다.
- 감각적 경험을 제공하는 콘텐츠: 교향곡 연주, 예술 작품 등과 같이 말로 설명하기 어려운 감각적 경험을 제공하는 콘텐츠는, 콘텐츠의 이름과 가능한 경우 간략한 설명만 제공할 수 있다.
- CAPTCHA: 사람임을 증명하기 위한 테스트(예: CAPTCHA)에는 사람이 읽을 수 있으나 컴퓨터가 쉽게 해독할 수 없는 콘텐츠가 포함된다. 이 경우 대체 텍스트를 제공하면 CAPTCHA의 목적이 무력화될 수 있으므로, CAPTCHA의 목적과 사용 방법을 간단히 설명하는 텍스트만 제공하여 사용자가 테스트의 의도를 이해할 수 있도록 한다.
- 사용자 생성 콘텐츠: 사용자가 콘텐츠를 업로드할 때, 대체 텍스트를 입력할 수 있는 입력란을 제공하여 해당 콘텐츠의 내용을 제공할 수 있도록 유도한다.

### 적용 사례

#### 1)	텍스트를 포함한 텍스트가 아닌 콘텐츠
이미지, 아이콘, 그래프, 오디오와 같은 텍스트가 아닌 콘텐츠에 텍스트를 포함하여 제공할 때, 해당 콘텐츠가 전달하는 의미나 용도를 모든 사용자가 동등하게 이해할 수 있도록 대체 텍스트를 제공해야 한다.

##### a)	짧은 텍스트가 포함된 이미지
다음은 이미지 내부에 짧은 텍스트가 포함된 사례이다. 이 경우, 시각적으로 전달되는 정보와 동등한 수준의 정보를 대체 텍스트로 제공해야 한다.

img 요소의 alt 속성을 활용하여 이미지에 포함된 텍스트와 동일한 내용을 제공하면, 화면을 볼 수 없는 사용자도 보조 기술을 통해 이미지가 전달하는 내용을 동등하게 전달받을 수 있다.

<figure>
  ![](/images/image004.png)
  <figcaption>이미지. 짧은 텍스트를 포함한 이미지 배너 사례 (출처: 정부24)</figcaption>
</figure>

```html
<a href="...">
	<img src="" alt="정부24를 사칭하여 개인정보를 요구하는 스미싱문자 및 피싱사이트에 각별히 주의하세요.">
</a>
```

##### b)	긴 내용의 텍스트가 포함된 이미지

이미지 내부에 긴 텍스트가 포함된 경우, 주요 정보를 동등하게 전달해야 한다. 이때 사용자가 내용을 쉽게 이해할 수 있도록 표, 목록, 문장 형태로 구조화하여 제공할 수 있다. 이때, 콘텐츠의 맥락과 목적을 고려하여 정보 전달 방식을 결정해야 한다.

다음은 이미지 내부에 긴 텍스트가 포함된 경우의 사례이다. 화면낭독프로그램은 img 요소의 alt 속성을 한 번에 읽기 때문에, 사용자는 내용을 단계적으로 탐색하거나 필요한 부분만 선택적으로 읽는 것이 어렵다. 따라서 img 요소의 alt 속성만으로 긴 내용을 전달하는 것은 한계가 있다.

이러한 한계를 해결하기 위해, 숨김 텍스트를 활용하거나 별도의 상세 설명 페이지 링크를 제공하는 등 대체 방법을 고려할 수 있다.

<figure>
  ![](/images/image005.png)
  <figcaption>이미지. 긴 텍스트를 포함한 이미지 배너 사례 (출처: 국민신문고)</figcaption>
</figure>

```css
.visually-hidden {
  clip-path: inset(100%);
  clip: rect(1px, 1px, 1px, 1px);
  height: 1px;
  overflow: hidden;
  position: absolute;
  white-space: nowrap;
  width: 1px;
}
```

```html
<img src="..." alt="국민신문고를 사칭하는 이메일에 주의하세요">
<p class="visually-hidden">
  신청한 적 없는 민원신청번호가 기재된 이메일은 열람하지 말고 삭제해주세요. <br>
  이메일을 열람한 경우 인터넷보호나라 등에 신고해주세요.(http://www.boho.or.kr)
</p>

```

#### 2)	텍스트가 포함되지 않은 텍스트가 아닌 콘텐츠
텍스트가 포함되지 않은 텍스트가 아닌 콘텐츠의 경우, 콘텐츠가 전달하는 의미나 기능을 인식할 수 있도록 적절한 대체 텍스트나 설명을 제공해야 한다.

##### a)	SVG 아이콘으로 구현된 버튼
아이콘은 정보를 간결하고 직관적으로 전달하는 시각적 기호이다. 따라서, 사용자가 아이콘의 의미를 쉽게 이해할 수 있도록 간결하면서도 명확한 대체 텍스트를 제공해야 한다.

다음은 검색 기능을 나타내는 돋보기 아이콘을 SVG로 적용한 사례이다. 이 경우, 아이콘이 전달하는 ‘검색’이라는 의미를 대체 텍스트로 제공해야 한다. 숨김 텍스트, aria-label, SVG의 title 요소를 활용하여 대체 텍스트를 제공한 예시이다.

<figure>
  ![](/images/image006.png)
  <figcaption>이미지. 검색을 표현한 돋보기 SVG 아이콘 사례 (출처: 다음)</figcaption>
</figure>

```html
<!-- 1.숨김 텍스트로 제공 -->
<button class="btn_search">
  <span class="visually-hidden">검색</span>
	<span class="ico_sch">
    <!-- IE에서 svg가 포커스 받는 것 방지가 필요한 경우focusable 속성 사용 -->
    <svg aria-hidden="true" focusable="false" width="28" height="28" viewBox="0 0 28 28" …>
		...
		</svg>
  </span>
</button>

<!-- 2.aria-label로 제공 -->
<button class="btn_search" aria-label="검색">
  <span class="icon_search">
		<svg focusable="false" width="28" height="28" viewBox="0 0 28 28"…>
		...
		</svg>
	</span>
</button>

<!-- 3.SVG 내 title 요소와 aria-labelledby로 제공 -->
<button class="btn_search">
	<span class="icon_search">
		<svg focusable="false" role="img" aria-labelledby="srch"  width="28" height="28" viewBox="0 0 28 28"…>
      <title id=”srch”>검색</title>
      ...
    </svg>
  </span>
</button>
```

이와 같이 SVG 아이콘에 대체 텍스트를 제공하는 방법은 여러 가지가 있다. 그러나 SVG 내 title 요소만으로 제공하는 방식은 iOS VoiceOver(iOS 18.1 기준)에서 이를 지원되지 않는다. 이를 보조 기술이 인식할 수 있도록 title 요소과 svg 요소를 aria-labelledby 속성으로 연결하여 보완할 수 있다.
이처럼 중요한 콘텐츠가 사용자에게 올바르게 전달되도록 하기 위해서는 보조 기술 지원 여부를 반드시 확인하고 테스트해야 한다.

### ■	개선 필요 사례
#### 1) `img`, `input type="image"` 등 이미지 요소에 alt 속성 미제공
다음은 배송 서비스를 홍보하기 위해 img 요소로 제공한 배너에 alt 속성이 제공되지 않은 사례이다. alt 속성이 누락된 경우, 화면낭독프로그램은 이미지 정보를 제대로 인식하지 못해 이미지의 URL 또는 파일명을 읽어 주거나, 단순히 ‘이미지’ 또는 ‘그래픽’이라고 낭독할 수 있다.

따라서 화면낭독프로그램 사용자는 이미지 정보를 인식할 수 없다.

<figure>
  ![](/images/image018.png)
  <figcaption>이미지. 정보가 있는 이미지 alt 속성이 누락된 사례</figcaption>
</figure>

```html
<div class="">
	<a href="..."><img src="bn1.jpg"></a>
</div>
<div class="">
	<a href="..."><img src=" bn2.jpg"></a>
</div>
```

##### 개선 방법
이미지 요소에는 alt 속성을 사용하여 이미지가 전달하는 정보와 동등한 수준의 대체 텍스트를 제공해야 한다.
```html
<img src="bn1.jpg" alt="이커머스 배송의 필수 파트너 000 당일배송">
<img src="bn2.jpg" alt="원하는 시간, 원하는 만큼 자유로운 배송 알바 000 라이더스">
```

#### 2)	의미 있는 배너 이미지에 대체 텍스트를 빈 값으로 제공
다음은 쇼핑 사이트에서 상품 주문 시 빠른 주문 방법을 안내하는 이미지를 제공하면서, 해당 이미지의 대체 텍스트를 빈 값으로 설정한 사례이다.

대체 텍스트가 빈 값(alt="")으로 제공된 이미지는 보조 기술에서 정보를 전달하지 않는 이미지로 인식되어, 사용자에게 이미지의 존재 여부조차 알리지 않는다. 이로 인해 화면을 볼 수 있는 사용자와 보조 기술 사용자 간에 인식하는 정보 차이가 발생한다.

<figure>
  ![](/images/image019.png)
  <figcaption>이미지. 빠른 주문 안내 텍스트 이미지에 대체 텍스트가 빈 값으로 제공된 사례</figcaption>
</figure>

```html
<img src="..." alt="">
```

##### 개선 방법
`alt="상품 주문 코드 2948-234를 이야기 해주시면 빠르게 주문이 가능합니다!"` 와 같이 이미지가 전달하는 정보를 `alt` 속성에 제공한다.

```html
<img src="..." alt="상품 주문 코드 2948-234를 이야기 해주시면 빠르게 주문이 가능합니다!">
```

#### 3)	이미지 정보와 다른 alt 속성의 대체 텍스트 내용
이미지가 전달하는 정보와 대체 텍스트의 내용이 서로 상이하게 제공된 사례이다.

이미지의 실제 내용은 ‘투명성을 위한 환경정보공개시스템’이지만 alt 속성이 ‘esg-green’으로 설정되어 보조 기술 사용자는 다른 정보를 제공받게 된다.

<figure>
  ![](/images/image020.png)
  <figcaption>이미지. 배너 이미지의 대체 텍스트가 상이한 사례</figcaption>
</figure>

```html
<img src="..." alt="esg-green">
```

##### 개선 방법
`alt="투명성을 위한 환경정보공개시스템"`과 같이 이미지가 전달하는 정보와 동등한 대체 텍스트를 제공한다.

```html
<img src="..." alt="투명성을 위한 환경정보공개시스템">
```

#### 4)	꾸밈 이미지에 대체 텍스트 제공
채용 공고 페이지에서 섹션 제목을 꾸미기 위해 사용된 아이콘 이미지의 사례이다.

이 아이콘은 단순한 꾸밈 요소임에도 불구하고 ‘icon’이라는 대체 텍스트가 제공되어, 보조 기술 사용자에게 불필요한 정보를 전달하고 있다.

예를 들어, 화면낭독프로그램은 이를 ‘icon, 이미지, 공통자격요건’과 같이 읽어 주므로, 사용자가 해당 이미지의 대체 텍스트가 잘못 작성되어 중요한 정보가 누락된 것인지, 제대로 제공된 것인지 판단하기 어렵게 된다.

<figure>
  ![](/images/image021.png)
  <figcaption>이미지. 꾸밈 용도로 사용된 이미지에 대체 텍스트가 제공된 사례</figcaption>
</figure>

```html
<td><img src="..." alt="icon">공통자격요건</td>
```

##### 개선 방법
이미지가 정보를 전달하지 않고 단순한 꾸밈 용도로 사용된 경우, alt=""와 같이 빈 값을 제공하여 보조 기술이 이미지를 무시하도록 설정해야 한다.

```html
<td><img src="..." alt="">공통자격요건</td>
```

#### 5)	관계 정보를 알 수 없는 대체 텍스트
다음은 기업의 조직도를 이미지로 제공한 사례이다. 조직도는 부서 간의 소속과 계층 관계 정보를 시각적으로 제공하지만, 다음 예시처럼 대체 텍스트가 이를 충분히 전달하지 않으면 보조 기술 사용자는 정보를 제대로 파악할 수 없다.

<figure>
  ![](/images/image022.png)
  <figcaption>이미지. 계층 구조로 이루어진 조직도 이미지 사례</figcaption>
</figure>

```html
<img src="..." alt="안전보건 경영을 추진하는 조직도입니다. 대표이사 산하의 경영지원부문 총무팀과 산업안전보건위원회를 운영 및 관리하고 있습니다.">
```

##### 개선 방법
조직도의 소속이나 상하 관계를 명확히 표현하여 계층 구조를 이해할 수 있도록 설명하는 텍스트 정보를 제공해야 한다. 

설명이 길다면 간략한 대표 설명을 alt 속성에 제공하고, 상세 설명은 별도의 숨김 텍스트로 제공할 수 있다. 또는, figure와 figcaption 요소를 활용하거나 별도의 텍스트 영역을 통해 설명을 제공할 수 있다.

```html
<img src="..." alt="조직도">
<div class="visually-hidden">
	대표이사 아래에는 경영지원부문과 산업안전보건위원회가 있습니다. 경영지원부문에서는 산업 안전 대응 총괄, 주관 운영 업무를 담당하고, 산업안전보건위원회에서는 안전보건사항 논의, 심의의결, 협력 업무를 담당합니다. 경영지원부문 아래에는 총무팀이 있고, 안전/보건관리 업무 및 시설 관리를 담당합니다.
</div>
```

#### 6)	img의 대체 텍스트를 title 속성만으로 제공
다음은 ‘지원하기’ 버튼을 이미지로 제공하면서, 대체 텍스트를 title 속성으로만 제공한 사례이다.

화면낭독프로그램 지원 또는 화면낭독프로그램의 설정에 따라 title 속성이 읽히지 않을 수도 있으며, 이로 인해 이미지 정보가 사용자에게 제대로 전달되지 않을 가능성이 높다.

<figure>
  ![](/images/image023.png)
  <figcaption>이미지. 이미지 요소의 대체 텍스트가 title 속성으로만 제공된 예시</figcaption>
</figure>

```html
<img src="apply_btn.jpg" title="지원하기">
```

##### 개선 방법
이미지의 대체 텍스트는 alt 속성을 사용하여 제공해야 하며, title 속성은 추가적인 정보 제공 용도로만 활용해야 한다.
```html
<img src="apply_btn.jpg" alt="지원하기">
```
#### 7)	정보가 포함된 이미지에 대체 텍스트 미제공
여러 장의 사진으로 구성된 뉴스 콘텐츠에서 모든 사진에 동일한 대체 텍스트를 제공한 사례이다. 이로 인해 화면낭독프로그램 사용자는 각 사진이 전달하는 개별적인 정보를 인식하지 못한다.

또한, title 속성에도 alt 속성과 동일한 내용이 제공되어 불필요한 중복이 발생하고 있다.

<figure>
  ![](/images/image024.png)
  <figcaption>이미지. 여러 개의 사진 이미지에 모두 동일한 대체 텍스트 제공 사례</figcaption>
</figure>

```html
<img src="MDE=_5.jpg" title="국립단체 청년 교육단원 통합 발대식" alt="국립단체 청년 교육단원 통합 발대식">
<img src="MDI=_6.jpg" title="국립단체 청년 교육단원 통합 발대식" alt="국립단체 청년 교육단원 통합 발대식">
<img src="MDQ=_6.jpg" title="국립단체 청년 교육단원 통합 발대식" alt="국립단체 청년 교육단원 통합 발대식">
```

##### 개선 방법
각 사진의 내용을 설명하는 개별적이고 의미 있는 대체 텍스트를 제공해야 한다. 또한, alt 속성과 동일한 title 속성은 불필요하므로 제거하여 중복된 정보가 전달되지 않도록 한다.
```html
<img src="MDE=_5.jpg" alt="국립예술단체 청년 교육단원 통합 발대식 단체 사진">
<img src="MDI=_6.jpg" alt="문화체육관광부 장관의 개회사 발표 전경">
<img src="MDQ=_6.jpg" alt="장관, 기관 단장이 참여한 토론회 진행 모습">
```

#### 11)	이모지로 잘못 된 대체 텍스트 제공
이모지를 아이콘처럼 사용하면서, 대체 텍스트 역시 이모지로 제공한 사례다. alt 속성값이 이모지로만 구성된 경우, 보조 기술 환경에 따라 이모지를 정상적으로 해석하지 못해 대체 텍스트가 없는 것으로 처리될 가능성이 있다.

따라서, 단순한 장식 목적이라면 alt=""(빈 값)를 사용하여 화면낭독프로그램이 불필요한 정보를 읽지 않도록 한다. 반면, 의미 전달이 필요한 경우에는 사용자가 쉽게 이해할 수 있는 문구로 직접 대체 텍스트를 제공해야 한다.

<figure>
  ![](/images/image030.png)
  <figcaption>이미지. 이모지를 아이콘으로 사용한 사례</figcaption>
</figure>


```html
<div>
  <span class="c-jUooxn">
    <img alt=" " class="c-hakyQ" src="1f4aa.webp">
	</span>
	<p>건강검진 양도 신청<span>복리 후생</span></p>
</div>
```

##### 개선 방법
위 사례에서는 이모지가 단순한 장식 용도로 사용되고 있으므로, alt 속성값을 빈 값(alt="")으로 설정하여 보조 기술에서 읽히지 않도록 한다.

참고로, 이모지는 환경에 따라 다르게 읽힐 수 있다. 예를 들어,   (알통) 이모지는 Windows에서는  "알통", iOS는 "근육질의 구부린 팔뚝", Android는 "이두박근"과 같이 플랫폼별로 다른 방식으로 전달된다. 따라서, 이모지를 의미 전달의 핵심 요소로 사용할 경우 플랫폼별 차이를 고려하여 적절한 대체 텍스트를 제공해야 한다.

```html
<div>
  <span class="c-jUooxn">
    <img alt="" class="c-hakyQ" src="1f4aa.webp">
	</span>
	<p>건강검진 양도 신청<span>복리 후생</span></p>
</div>
```

## 1.2 멀티미디어 대체수단

<Aside title="검사 항목">
멀티미디어 콘텐츠에는 자막, 대본 또는 수어를 제공해야 한다.
</Aside>

### 목적
멀티미디어 콘텐츠가 전달하는 모든 정보(대화, 음향 효과, 배경 소리 등)가 청각이나 시각 등 특정 감각에 제한이 있는 사용자에게도 동등하게 전달되도록 자막, 대본 또는 수어와 같은 동등한 대체 수단을 제공하는 것이다.

##	배경 및 필요성
청각에 제한이 있는 사용자는 멀티미디어 콘텐츠에 포함된 소리를 제대로 듣지 못하거나 전혀 들을 수 없으므로, 자막, 대본, 또는 수어와 같은 대체 수단을 제공하여 다른 감각을 통해 정보를 인식할 수 있도록 지원해야 한다.

자막과 대본은 콘텐츠의 오디오 트랙과 동기화되어야 하며, 대화뿐만 아니라 음향 효과, 음악 등 비언어적 소리 정보도 함께 전달해야 한다. 이를 통해 모든 사용자가 콘텐츠의 의미와 맥락을 온전히 이해할 수 있으며, 특히 청각 장애가 있는 사용자도 정보 접근에서 배제되지 않는다.

또한, 시각에 제약이 있는 사용자는 영상 내 문자 정보나 나레이션이 없는 안내 동영상을 인식하기 어려울 수 있다. 따라서 청각을 활용하거나, 대본 등 다른 감각을 통해 콘텐츠를 이해할 수 있도록 제공해야 한다.

##	기본 규칙
다음 중 한 가지 이상의 대체 수단을 제공한다.

#### 자막 제공
- 멀티미디어 콘텐츠의 오디오 정보를 화면에 자막으로 제공한다.
- 자막은 대화, 화자 식별, 음향 효과 등 모든 청각적 요소를 포함하며, 콘텐츠와 동기화되어야 한다.
- 자막 제공 방식은 열린 자막(Open Caption)과 닫힌 자막(Closed Caption)으로 구분된다. 사용자가 필요에 따라 자막을 켜거나 끌 수 있으며, 자막의 위치, 크기, 색상 등을 조정할 수 있는 닫힌 자막 방식을 제공하는 것이 더 유용하다.
- 소프트웨어 등에 의해 자동으로 생성된 자막은 정확도가 낮을 수 있으며, 중요한 배경 소리나 음악 등이 누락되거나 부정확하게 표현될 우려가 있다. 따라서 검토 및 보정 과정을 거쳐 제공해야 한다.

#### 대본 제공
- 멀티미디어 콘텐츠의 오디오 및 비디오 트랙에서 전달되는 모든 정보를 문자로 변환한 대본을 제공한다.
- 대본은 대화뿐만 아니라 설명, 배경 소리, 음악 등 모든 청각적 요소가 포함되어야 한다.
- 대본은 사용자가 쉽게 확인할 수 있도록 멀티미디어 콘텐츠 근처에 배치하며, 현재 재생 중인 부분을 식별할 수 있도록 표시해야 한다.
- 내용이 길 경우, 화면 이동 없이 계속 확인할 수 있도록 스크롤 가능한 형태로 제공하며, 재생 중인 위치가 화면 내에서 유지되도록 자동 스크롤 기능을 지원하는 것이 바람직하다.

#### 수어 제공
- 청각장애인 사용자가 멀티미디어 콘텐츠의 내용을 이해할 수 있도록, 수어를 통해 정보를 전달한다. 수어는 청각 장애인 사용자들에게 가장 익숙하고 자연스러운 언어이므로, 복잡하거나 전문적인 정보가 포함된 콘텐츠에서 특히 효과적이다.
- 콘텐츠 화면 내에서 수어 통역사가 명확히 보이도록 별도의 공간을 배치한다.
- 수어 화면은 콘텐츠의 주요 장면이나 텍스트를 가리지 않도록 배치하여 중요한 정보가 손실되지 않도록 해야 한다.

소리 없이 영상만 제공되는 콘텐츠에도 동일한 대체 수단이 필요하다.

시각에 제약이 있는 사용자는 멀티미디어 콘텐츠에 포함된 문자 정보나 영상을 인식하기 어려우므로, 청각 등 다른 감각을 활용할 수 있는 대체 정보를 제공해야 한다

#### 대본 제공
- 청각을 사용하는데 제한이 있는 사용자를 위한 대본과 마찬가지로, 문자로 변환된 대본은 화면낭독프로그램을 통해 접근할 수 있어야 한다.
- 시각적 요소가 중요한 경우 해당 장면에 대한 설명도 포함하여, 사용자가 콘텐츠의 전체적인 맥락을 이해할 수 있도록 돕는다.

#### (권고) 나레이션 또는 화면 해설 제공
- 시각적 정보를 인지하기 어려운 사용자를 위해, 멀티미디어 콘텐츠의 시각적 정보를 음성으로 설명하는 나레이션 또는 화면 해설을 제공한다.
- 화면 해설은 영상에서 전달하고자 하는 주요 시각적 정보를 설명해야 하며, 등장인물의 행동, 표정, 의상, 장면의 변화 등 대사만으로 이해하기 어려운 요소를 포함해야 한다.
- 원래의 음향을 방해하지 않도록, 대사나 중요한 음향 효과가 없는 부분에 삽입해야 한다.
- 콘텐츠의 시각적 정보와 동기화되도록 적절한 타이밍으로 제공해야 한다.
- 화면 해설을 제공하는 방식으로는 기본 음성 트랙에 화면 해설을 포함하는 통합 방식과 별도의 음성 트랙으로 제공하는 선택적 방식이 있다. 사용자가 필요에 따라 화면 해설을 켜고 끌 수 있는 선택적 방식을 권장한다.

### 용어
<dl>
	<dt>멀티미디어 콘텐츠</dt>
	<dd>오디오, 비디오, 이미지 등 여러 미디어 요소를 동기화하여 제공함으로써, 시간에 따라 변화하는 정보를 전달하는 콘텐츠를 의미한다. 이러한 콘텐츠에는 동영상이나 오디오와 같은 시간 기반 미디어가 포함되며, 재생 중 특정 시점에서 사용자의 상호작용(예: 재생, 일시정지, 특정 지점 이동)이 요구될 수 있다.</dd>
	<dt>오디오트랙(audiotrack)</dt>
	<dd>멀티미디어 콘텐츠에서 소리를 담고 있는 독립적인 부분을 의미한다. 음성, 음악, 효과음 등이 포함되며, 하나의 멀티미디어 콘텐츠에 여러 개의 오디오 트랙을 포함할 수 있다.</dd>
	<dt>동기화(synchronized)</dt>
	<dd>멀티미디어 콘텐츠에서 서로 다른 유형의 미디어 요소(예: 오디오, 비디오, 자막 등)가 시간적으로 일치하도록 조정되는 것을 의미한다. 예를 들어, 영화에서 배우의 대사(오디오)와 입술 움직임(비디오)이 정확히 맞아떨어지는 것이 동기화의 한 예이다.</dd>
	<dt>개방 자막 또는 열린 자막(Open Caption)</dt>
	<dd>영상에 고정되어 인코딩된 자막으로, 재생 시 항상 표시되는 자막을 의미한다. 이는 영상의 한 부분으로 포함되어 있어, 사용자가 자막을 켜거나 끌 수 없고, 개별적으로 조정할 수도 없다.</dd>
	<dt>폐쇄 자막 또는 닫힌 자막(Closed Caption)</dt>
	<dd>영상과 별도로 제공되어 사용자가 필요에 따라 켜거나 끌 수 있는 자막을 말한다. 자막 데이터가 영상과 분리되어 있어, 플레이어나 시스템 기능에 따라 자막의 위치, 크기, 글꼴, 색상 등을 사용자가 자유롭게 조정할 수 있다.</dd>
	<dt>화면 해설</dt>
	<dd>시각적 정보를 보조하는 별도의 음성 설명으로, 대사나 소리만으로 이해하기 어려운 장면(등장인물의 행동, 표정, 의상, 장면 변화 등)을 설명하는 기능을 한다. 이는 기존 대사나 중요한 소리를 방해하지 않는 구간에 제공된다.</dd>
</dl>

### 적용 사례

#### 1)	멀티미디어 콘텐츠 정보의 동등한 자막 제공
멀티미디어 콘텐츠 내 대화, 의미 있는 음향 효과를 포함하여 소리를 통해 전달되는 비언어적 정보 등을 동등하게 이해할 수 있도록 자막을 제공한다.

이때, 자막은 실시간으로 내용을 파악할 수 있도록 동기화되며, 영상과 시각적으로 명확히 구분되도록 제공해야 한다. 대화가 포함된 경우, 화자의 식별 정보(예: “길동: 안녕하세요”)를 제공하여, 여러 사람이 대화하는 상황에서도 각 화자의 발언을 명확히 구분할 수 있도록 한다.
##### a)	말하는 사람을 식별할 수 있고 실시간으로 동기화된 대화 제공
여러 사람이 대화하는 상황에서 말하는 사람이 화면에 나타나지 않거나 여러 사람이 동시에 등장하는 경우, 대화 내용의 주체를 알기 어렵다.

이때, 말하는 사람을 함께 표시하여 자막을 제공하면, 사용자가 대화 내용을 더욱 정확하게 이해할 수 있다.

이때, 자막에 의미 있는 음향 효과를 추가로 제공하면, 사용자가 장면이 전달하고자 하는 맥락과 상황을 더욱 명확히 이해할 수 있다.

데이터 표에서 행과 열의 제목(제목 셀)과 내용(내용 셀)을 연결하여, 보조 기술 사용자도 복잡한 표 형식 데이터를 쉽게 이해하고 탐색할 수 있도록 돕는 항목이다.

##	배경 및 필요성
화면낭독프로그램과 같은 보조 기술 사용자는 HTML의 선형 구조와 표의 구조를 기반으로 데이터를 순차적으로 탐색한다. 이때, 제목 셀과 내용 셀의 관계가 명확하게 정의되지 않으면, 데이터 간의 연관성을 파악하기 어렵고, 특정 셀이 어떤 제목 셀과 연결되는지 이해하기 어려워진다.

특히 보조 기술 사용자는 표를 인식하는 방식에서 일반 사용자와 차이가 있다. 화면낭독프로그램은 표를 읽을 때 다음과 같은 순서와 방법으로 데이터를 탐색한다.

- 표의 셀은 마크업 순서에 따라 순차적으로 읽히며, 사용자는 각 셀의 내용을 듣고 해당 셀이 어느 행과 열에 속하는지 위치 정보를 전달 받는다.
- 특정 내용 셀을 읽을 때, 화면낭독프로그램이 해당 셀이 속한 열이나 행의 제목 정보를 함께 읽어 주어 사용자가 맥락을 파악할 수 있도록 한다.
- 화면낭독프로그램에는 표의 행과 열을 따라 셀 간 이동을 돕는 단축키가 있어 탐색을 지원하며, 한 페이지에 여러 개의 표가 있을 경우 표 단위로 탐색할 수 있다.

만약 제목 셀과 내용 셀의 관계가 모호하거나 표 구조에 대한 추가 정보가 부족하다면, 화면낭독프로그램은 내용을 순차적으로 읽기 때문에 표가 가진 맥락과 의미를 제대로 전달하기 어려워진다.

##	기본 규칙
- 표는 table, caption, tr, th, td 등 표 전용 HTML 요소를 올바르게 사용한다.
- 제목 셀에는 th 요소를 사용한다. 복잡한 표에서도 내용 셀이 어떤 제목 셀과 연결되는지 알 수 있도록 scope 속성이나 headers, id 속성을 함께 설정한다.
- 표에는 caption 요소를 사용해 제목과 설명 정보를 제공한다. 이를 통해 화면낭독프로그램 사용자는 표 내용을 일일이 읽지 않아도 전체 맥락과 요점을 빠르게 파악할 수 있다.
- 레이아웃(화면 배치) 목적으로 테이블을 사용하는 것은 가급적 지양한다. 불가피한 경우, 의미 부여를 위한 th, caption 요소 및 summary, scope 속성은 사용하지 않는다.

  - 다중 데이터 차원 표현(교차 표)
  - 중첩 테이블(nested table) 구조
  </dd>
</dl>

### 적용 사례

#### 1)	표 정보 제공
보조 기술이 표의 용도와 목적을 쉽게 파악할 수 있도록, 적절한 제목과 설명을 제공해야 한다.
##### a)	표의 제목 제공
표의 제목은 caption 요소를 사용하여 제공한다. caption에 작성하는 내용은 표의 용도와 목적을 전달해야 하며, 사용자가 표의 정보를 이해할 수 있도록 간결하고 직관적으로 작성해야 한다.

반면에, 표 상단에 h3 요소나 일반 텍스트로 제목을 배치하더라도, 보조 기술은 이를 표의 제목으로 인식하지 않는다. 따라서, 제목이 시각적으로 표시되었더라도, caption 요소를 제공해 보조 기술 사용자도 표의 내용을 빠르게 이해할 수 있도록 해야 한다.

```html
<!-- 좋은 예시 -->
<table>
<caption>2024년 1분기 부서별 매출 현황</caption>
  <thead>
    <tr>
<th scope="col">부서</th>
        <th scope="col">매출액</th>
    </tr>
  </thead>

<!-- 잘못된 예시 -->
<h2>2024년 1분기 부서별 매출 현황</h2>
<table>
<thead>
    <tr>
      <th scope="col">부서</th>
      <th scope="col">매출액</th>
    </tr>
</thead>
```
#### b)	표의 설명 제공
표가 복잡하거나 특이한 구조를 갖는 경우, 간단한 요약 설명을 통해 어떤 행이나 열에서 어떤 내용을 찾을 수 있는지 안내하면 사용자가 표를 빠르게 이해할 수 있다.

HTML 5에서는 `summary` 속성이 지원되지 않으므로, `caption` 요소에 설명을 함께 작성하거나, `aria-describedby` 속성을 사용하여 설명 텍스트와 연결하거나, `figure`와 `figcaption` 요소를 활용하는 등의 대체 기법을 사용할 수 있다.

다음은 `caption` 요소에 요약 정보를 추가하여, 제목 셀이 2행으로 구성되고 일부가 병합된 복잡한 표를 화면낭독프로그램 사용자가 이해할 수 있도록 설명을 제공한 사례이다.

<figure>
  ![](/images/image045.png)
  <figcaption>이미지. 제목 셀 구조가 복잡한 테이블 사례 (출처: 서울특별시-서울재정포털)</figcaption>
</figure>

```html
<table>
	<caption>
		<strong>세입현황표</strong><br>
		<p>
			세입과목, 전입누계, 기간중 수입액(수입액, 과오납 반환, 과목경정, 차액), 금일누계로 구성되어있습니다.
		</p>
	</caption>
...
```
### 2)	표 제목 셀과 내용 셀의 연결
보조 기술이 표에서 제목 셀과 내용 셀 간의 관계를 이해할 수 있도록 구현해야 한다. 이를 통해 화면낭독프로그램이 각 셀이 어떤 제목 셀과 연관되는지 올바르게 읽어 주어, 사용자가 표를 보다 쉽게 이해할 수 있도록 돕는다.
#### a)	첫 행이 제목 셀로 구성된 표
첫 번째 행 전체가 제목 셀(th)로 구성된 기본적인 형태의 표라면, 화면낭독프로그램이 추가적인 접근성 속성(scope, id와 headers)을 생략하여도 구조를 전달할 수 있다.

<figure>
  ![](/images/image046.png)
  <figcaption>이미지. 첫 행이 제목으로 제공된 표와 코드 사례 (출처: 국민건강보험)</figcaption>
</figure>

```html
<table>
<caption>지역 임시 진료 안내</caption>
  <tr>
    <th>월</th>
    <th>일</th>
    <th>지역명</th>
    <th>대상</th>
    <th>의료진</th>
  </tr>
```

#### b)	첫 열이 제목 셀로 구성된 표
첫 열이 제목 셀인 경우, th 요소에 scope=”row”를 사용한다. 이를 통해 보조 기술은 해당 열이 행 제목임을 인식하고, 내용 셀이 어떤 행에 속하는지 전달할 수 있다.

<figure>
  ![](/images/image047.png)
  <figcaption>이미지. 첫 열이 제목으로 제공된 표와 코드 사례 (출처: 한국장학재단)</figcaption>
</figure>

```html
<table>
<caption>대학생 재능봉사 캠프 사업의 연도별 참여 대학 현황</caption>
  <tr>
<th scope="row">2010학년도 <br> (22개교)</th>

  </tr>
  <tr>
    <th scope="row">2011학년도 <br> (32개교)</th>
    <td>강원대학교, 건국대학교, 경북대학교, 경상대학교, ...</td>
  </tr>
<tr>
    <th scope="row">2012학년도 <br> (45개교)</th>
    <td>강원대학교, 건국대학교, 경북대학교, 경상대학교, ...</td>
  </tr>
</table>
```

#### c)	제목 셀이 두 줄 이상이거나 병합된 복잡한 표 
하나의 행이나 열에 두 개 이상의 제목 셀이 포함된 표에서는, 각 제목 셀에 고유한 ID를 지정해야 한다. 제목 셀은 `<th id="고유ID값">` 형식을 사용하고, 내용 셀은 `<td headers="연결이 필요한 제목 셀의 id값">` 형식으로 연결한다.

아래 예시는 2개의 열이 각각 제목 역할을 하는 주가 정보 표로, 제목 셀의 ID를 내용 셀의 headers 속성으로 연결한 사례다.

```html
<tr>
  <th id="prev-index">전일지수</th>
  <td headers="prev-index">727.66</td>
```

### 개선 필요 사례

#### 1)	표의 제목 미제공
다음은 인터넷 상품을 설명하는 표에서 표 제목이 제공되지 않은 사례다. 화면낭독프로그램 사용자는 표의 주제나 내용을 탐색하기 전에 이를 파악하기 어려워 어떤 정보가 제공되는지 알 수 없다. 특히 한 페이지 내 표가 여러 개 있을 경우 각각의 표를 구분하거나 맥락을 이해하는 데 어려움을 겪게 된다.

<figure>
  ![](/images/image049.png)
  <figcaption>이미지. 표의 제목이 제공되지 않은 사례</figcaption>
</figure>

```html
<div class="title_block">인터넷 구분</div>
<table>
  <tbody>
    <tr>
      <th>구분</th>
      <th>속도</th>
      <th>비고</th>
    </tr>
...
```

##### 개선 방법
표 상단에 제목 텍스트를 제공하더라도, 이는 HTML 구조에서 표의 제목으로 인식되지 않는다. 따라서 caption 요소를 사용하여 표의 제목을 명확히 제공해야 한다.

```html
<div class="title_block">인터넷 구분</div>
<table>
<caption>인터넷 상품 안내</caption>
  <tbody>
...
```

#### 2)	표 제목을 display:none 으로 숨김
다음은 표 제목을 caption 요소로 제공하였지만, CSS에서 `display:none`으로 설정되어 화면뿐만 아니라 보조 기술에도 해당 내용이 전달되지 않는 사례이다.

<figure>
  ![](/images/image050.png)
  <figcaption>이미지. 표의 제목 정보를 포함한 caption 요소를 display:none으로 노출되지 않도록 설정한 예시</figcaption>
</figure>

```css
caption {
    display: none;
}
```

```html
<table border="1">
<caption>게시판 목록</caption>
<colgroup>...</colgroup>
  <thead>
      <tr>
          <th scope="col">번호</th>
          <th scope="col">제목</th>
          <th scope="col">카테고리</th>
          <th scope="col">작성자</th>
          <th scope="col">작성일</th>
          <th scope="col">조회</th>
          <th scope="col">평점</th>
      </tr>
  </thead>
  ...
```

##### 개선 방법
표의 제목이나 설명은 화면에 보이지 않더라도 보조 기술에서 접근할 수 있어야 한다.

`display:none`, `visibility:hidden`, `font-size:0` 등으로 숨기면 보조 기술에서도 해당 콘텐츠를 인식하지 못하므로 이러한 방식은 피해야 한다. 숨김 처리를 할 때는 보조 기술에서 올바르게 탐색할 수 있도록 적절한 CSS 기법을 사용하여 제공해야 한다.

앞서 다룬 적절한 대체 텍스트 제공 검사항목에서 제시한 `.visually-hidden` CSS 정의를 참조하여 보조 기술에서 인식할 수 있도록 구현한다.

#### 3)	부적절한 표 제목
객실 정보 표에 ‘지리산 버스 투어 정보’라는 잘못된 표 제목이 적용된 사례다. 표의 제목과 실제 제공되는 정보가 다르기 때문에, 화면낭독프로그램 사용자는 표의 내용을 해석하는 데 혼동을 겪을 수 있다.

<figure>
  ![](/images/image051.png)
  <figcaption>이미지. 객실 정보 표에 ‘지리산 버스 투어 정보’로 제목이 잘못 적용된 예시</figcaption>
</figure>

```html
<caption>지리산 버스 투어 정보</caption>
```

##### 개선 방법
표의 제목은 표가 제공하는 내용을 명확히 이해할 수 있도록 ‘객실 정보’ 로 제공한다.

```html
<caption>객실 정보(유형별 기준 인원 및 1박 판매가)</caption>
```

#### 4)	표 설명 미제공
다음은 여행 정보를 제공하는 웹 페이지에서 대상 도시의 정보를 표로 제공한 사례이다.

표 제목으로 ‘도시 정보’가 제공되었지만, 이는 표의 내용을 충분히 전달하지 못한다. 표 제목이 제공되었음에도 화면을 볼 수 없는 사용자는 이 표가 어떤 정보를 전달하는지, 각 열과 행이 무엇을 의미하는지 예측하기 어렵다.

<figure>
  ![](/images/image052.png)
  <figcaption>이미지. 제목만으로는 표의 내용을 충분히 예측하기 어려운 예시</figcaption>
</figure>

```html
<table>
	<caption>도시 정보</caption>
	...
```

##### 개선 방법
summary 속성에 해당 표에 대한 정보를 추가하여 ‘주요 여행 도시의 현지 시간, 환율, 콘센트 정보 안내’와 같이 간결하게 설명을 함께 작성할 수 있다. 

```html
<table summary="도시의 현재 시간, 환율, 콘센트 정보">
	<caption>도시 정보</caption>
	...
```

앞서 설명한 것처럼, HTML5에서는 summary 속성이 제거되었으므로, caption 요소에 설명을 포함하거나 aria-describedby 속성을 이용하여 하는 등 다양한 대체 방법을 통해 보조 기술에 정보를 전달할 수 있다.

```html
<p id="tbl-cityinfo">각 도시의 현재 시간, 환율, 콘센트 정보</p>
<table aria-describedby="tbl-cityinfo">
	<caption>도시 정보</caption>
	...
```

#### 5)	제목 셀 미제공
콘서트 정보 표에서 시각적으로는 제목 행이 보이지만, 마크업에서 해당 행에 th 요소를 사용하지 않아 보조 기술이 이를 제목 셀로 인식하지 못하는 사례다. 이로 인해 화면낭독프로그램 사용자는 제목 셀과 내용 셀 간의 정보 연관성을 파악하기 어려워지고, 특정 정보를 빠르게 찾거나 표 전체를 이해하는데 어려움을 겪을 수 있다.

<figure>
  ![](/images/image053.png)
  <figcaption>이미지. 콘서트 일정 표 예시</figcaption>
</figure>

```html
<table>
<caption>2024년 12월 콘서트 정보</caption>
  <tr>
    <td>날짜</td>
    <td>콘서트 이름</td>
    <td>가수</td>
  </tr>
  <tr>
    <td>2024-12-05</td>
    <td>2024 월드투어 - 서울</td>
    <td>아이브 (IVE)</td>
</tr>
...
```

##### 개선 방법
첫 행이 제목 셀인 경우, 제목 셀(th 요소)에 scope 속성을 사용하여 `<th scope="col">` 형식으로 제공하고 제목 행 전체는 thead 요소를 함께 사용하면 구조를 더 명확히 할 수 있다.  

제목인 첫 행에 th 요소를 부여하고, 제목 행 전체를 thead 요소로 감싸면 구조를 더욱 명확히 표현할 수 있다.

```html
<table>
  <caption>2024년 12월 콘서트 정보</caption>
  <thead>
    <tr>
      <th>날짜</th>
      <th>콘서트 이름</th>
      <th>가수</th>
    </tr>
  </thead>
	<tbody>
	<tr>
      <td>2024-12-05</td>
      <td>2024 월드투어 - 서울</td>
      <td>아이브 (IVE)</td>
    </tr>
		...
```

#### 6)	표 안에 표 제공 (중첩된 테이블)
지역별 축제 정보를 담은 표에서, 내용 셀에 기념품샵 정보를 추가로 제공하기 위해 중첩된 테이블을 사용한 사례다.

중첩 테이블은 시각적으로는 구분하기 쉽지만, 순차적으로 탐색하는 보조 기술 입장에서는 표의 계층 구조가 복잡해 질 수 있다. 예를 들어, 화면낭독프로그램은 표의 행, 열, 셀에 대해 정보를 제공하지만, 중첩된 표가 있으면 제목 셀과 내용 셀 사이의 관계를 정확히 해석하지 못할 수 있다. 

또한, 셀 간 이동 방식이 예상과 달라져 탐색 명령이 의도대로 작동하지 않을 가능성이 높아진다.

<figure>
  ![](/images/image054.png)
  <figcaption>이미지. 지역별 축제 정보 표 안에 기념품샵 정보 표가 포함된 예시</figcaption>
</figure>

##### 개선 방법
가능하면 중첩 구조를 없애고 하나의 표로 표현한다. 하나의 표로 제공하기 어려운 경우, 중첩된 표를 분리하여 제공하거나 목록 요소 등을 활용할 수 있다.

#### 7)	레이아웃 용도로 사용된 표에 제목, 설명, 제목 셀 사용
다음은 도서전 이벤트 페이지에서 콘텐츠 배치를 위한 레이아웃 용도로 table 요소를 사용한 사례이다.

데이터 표가 아님에도 표 제목이나 설명을 제공하면, 화면낭독프로그램이 이를 데이터 표로 인식하여 불필요한 설명을 제공하고 사용자는 해당 테이블을 데이터 표로 오해할 가능성이 있다. 이로 인해 사용자는 콘텐츠를 정확히 이해하지 못하거나 불편을 겪게 된다.

<figure>
  ![](/images/image055.png)
  <figcaption>이미지. 콘텐츠를 배치하는데 사용된 표 사례</figcaption>
</figure>

```html
<table>
<caption>도시전 2024 안내</caption>
<tbody>
    <tr>
      <th>도시전 2024</th>
    </tr>
    <tr>
      <td><img src="..." alt="..."></td>
    </tr>
    <tr>
      <td><p>"도시전 2024"는 12월 30일부터 12월 31일까지... </p><td>
    </tr>
    ...
  </tbody>
</table>
```

##### 개선 방법
콘텐츠 배치를 위한 레이아웃을 구현할 때는 table 요소 대신 적절한 HTML 요소와 CSS를 사용해 구조화하는 것이 바람직하다. 불가피하게 table 요소를 사용해야 하는 경우(예: 메일 발송 전용 콘텐츠 등), 데이터 표로 인식되지 않도록 th, caption 요소 및 summary, scope 속성과 같이 의미를 가지는 정보는 제공하지 않으며, 가능한 간단하게 작성해야 한다.

```html
<table>
	<tr>
    <td>도시전 2024</td>
  </tr>
  <tr>
    <td><img src="..." alt="..."></td>
  </tr>
  <tr>
    <td><p>"도시전 2024"는 12월 30일부터 12월 31일까지... </p><td>
  </tr>
	...
</table>
```

## 1.4 명료성

<Aside title="검사 항목">
콘텐츠는 색에 관계없이 인식될 수 있어야 한다.
</Aside>

### 목적
웹 콘텐츠에서 제공하는 모든 정보가 색상에만 의존하지 않도록 설계하여, 색각 이상 사용자, 흑백 디스플레이 사용자, 고대비 모드를 사용하는 사용자 등 색을 구별하는 데 어려움이 있는 환경에서도 정보를 올바르게 인식할 수 있도록 보장한다.

##	배경 및 필요성
웹 콘텐츠에서 선택된 메뉴, 오류 표시, 필수 입력 등의 정보를 색상만으로 전달하는 경우, 색상을 구별하기 어려운 사용자는 중요한 정보를 놓칠 가능성이 높다.

예를 들어, 입력 폼에서 필수 입력란을 빨간색으로만 표시하거나, 선택된 상태를 텍스트 색상 변경만으로 표현한 경우, 색상을 구별하기 어려운 사용자는 해당 정보를 인식하지 못할 수 있다. 따라서 텍스트, 아이콘, 무늬 등 추가적인 시각적 요소를 함께 사용하여 정보를 전달해야 한다.

<figure>
	![](/images/image075.png)
	<figcaption>이미지. 녹색과 빨간색으로만 구분된 유효성 검사 결과(왼쪽)와 화면의 색을 제거(오른쪽)하여 비교한 예시</figcaption>
</figure>

차트, 그래프 등에서 색상을 정보 전달의 유일한 수단으로 사용해서는 안 된다. 대신, 무늬, 텍스트, 아이콘, 선 굵기 등의 추가적인 시각적 요소를 함께 제공하면, 색각 이상 사용자뿐만 아니라 고대비 모드 및 흑백 디스플레이 환경에서도 콘텐츠의 의미를 명확히 전달할 수 있다.

##	기본 규칙
- 색상만으로 내용을 구분하거나 정보를 제공하지 않고, 무늬나 텍스트 등의 보조적인 시각 정보를 함께 제공해야 한다.
- 차트, 그래프 등에서는 색상 외에도 패턴, 아이콘, 레이블 등을 추가하여 데이터를 구별할 수 있도록 한다.
- 고대비 모드, 흑백 디스플레이 환경에서도 콘텐츠의 의미와 구조를 명확히 전달할 수 있도록 디자인한다.

# 2. 운용의 용이성

운용의 용이성이란 사용자가 사이트 내 모든 콘텐츠를 손쉽게 탐색하고 조작할 수 있도록 지원하는 것이다. 특히, 키보드만으로도 모든 기능을 문제없이 사용할 수 있어야 한다. 예를 들어, 탭(Tab) 키를 이용해 논리적인 순서로 초점이 이동하도록 구성하고, 초점을 받은 상태에서 예기치 않게 새 창이 열리거나, 입력하자마자 자동으로 페이지가 이동하는 등의 ‘원치 않는 맥락 변화’를 방지하는 것도 중요하다.

또한, 키보드 사용이 어려운 제한적인 상황(예: 부상으로 인한 신체적 제한, 마우스 조작이 불편한 환경)에서도 사용을 보장해야 한다. 스마트폰이나 태블릿을 사용하는 사용자에게는 터치 제스처를 지나치게 복잡하게 만들지 않고, 음성 명령을 사용하는 환경도 고려해야 한다.

다음과 같은 환경을 생각해 볼 수 있다.
- 키보드만 사용해 게시물 작성, 댓글 입력이 가능하면, 한 손만 사용할 수 있는 사용자나 화면낭독프로그램을 이용하는 사용자도 서비스를 제약 없이 이용할 수 있다.
- 자동 슬라이드 배너에 ‘정지’ 버튼을 제공하면, 독서 속도가 느리거나 집중을 위해 움직임을 막고 싶은 사용자도 필요한 정보를 놓치지 않는다.
- 버튼에 마우스를 올렸을 때 즉시 팝업이 뜨는 대신, 클릭해야 열리도록 설계하면 오작동이 줄어들고 사용자의 예측 가능성이 커진다.
- 입력 서식에 초점이 이동했을 때, 시각적으로 초점이 어디에 있는지 표시해 주고, 필요하다면 오류 안내 텍스트 등을 제공하여 보조 기술이 서식 상태를 정확히 전달할 수 있도록 한다.

운용의 용이성을 확보하면, 사용자는 자신에게 적절한 속도와 방식으로 사이트를 이용할 수 있다.

## 2.1 입력장치 접근성

<Aside title="검사 항목">
모든 기능은 키보드만으로도 사용할 수 있어야 한다.
</Aside>

### 목적
웹 콘텐츠의 모든 기능을 키보드만으로도 사용할 수 있도록 보장하여, 장애가 있는 사용자뿐만 아니라 마우스 사용이 어려운 모든 사용자도 웹 콘텐츠를 원활하게 이용할 수 있도록 한다.

##	배경 및 필요성
웹 콘텐츠를 사용할 때, 마우스를 사용할 수 없는 사용자들은 키보드를 통한 접근이 필수적이다.

예를 들면, 시각장애로 인해 화면낭독프로그램을 사용하는 사용자, 지체장애로 인해 마우스 조작이 어려운 사용자, 손목 골절과 같은 일시적인 손상으로 인해 마우스 사용이 어려운 사용자 모두가 키보드를 통한 접근이 가능해야 한다.

키보드만으로 콘텐츠를 사용할 수 없는 경우, 이러한 사용자들은 웹 콘텐츠에 접근하지 못하거나 특정 기능을 사용할 수 없게 된다. 예로, 키보드로 조작할 수 없는 버튼, 메뉴, 폼 요소 등이 존재하면 서비스 이용이 불가능해질 수 있다.

또한, 키보드 조작을 선호하는 사용자도 키보드로 접근할 수 없는 웹 콘텐츠에서 동일한 불편과 제약을 경험한다. 장애가 없는 사용자라 하더라도 터치패드 고장과 같은 하드웨어 문제로 인해 일시적으로 마우스를 사용할 수 없는 상황이 발생할 수 있다.

##	기본 규칙
- 모든 상호작용 요소(링크, 버튼, 입력란 등)는 기본적으로 키보드 초점을 받을 수 있는 a, button, input 등의 HTML 기본 요소로 구현한다.
- 마우스 이벤트와 동일한 기능을 수행하도록, onclick 이벤트를 키보드 이벤트(keydown 또는 keyup)와 함께 처리한다.
- 커스텀 UI 컴포넌트(사용자가 정의한 버튼, 콤보박스, 스위치 등)는 키보드 초점 접근이 가능하도록 설정하고, 키보드 이벤트 핸들러를 추가하여 키보드로도 작동하도록 구현한다. 이때, 기본 HTML 요소의 키보드 동작 방식을 따라서 구현한다.
- 마우스를 올리면(hover) 작동하는 경우, 키보드 초점(focus)을 받았을 때도 동일한 기능이 수행되도록 구현한다.
- 실제 키보드만 사용하여 Tab/Shift+Tab 키로 웹 페이지 전반의 탐색, 입력, 상호작용을 테스트하고, 키보드 전용 사용자도 모든 기능에 접근할 수 있는지 확인한다.
- 키보드 함정(keyboard trap)이 발생하지 않도록 한다.

### 적용 사례

#### 1)	키보드 초점을 받을 수 있는 HTML 기본 요소 사용
사용자와 상호작용을 하는 링크, 버튼, 입력폼 등은 a, button, input 등 HTML 기본 요소로 구현한다. 이러한 요소들은 기본적으로 키보드 초점을 받을 수 있어 키보드 접근이 가능하며, Tab 키를 눌러 요소 간 이동이 가능하다.

```html
<!-- 상호작용 가능한 주요 HTML요소 예 -->
<a href="https://www.kioskui.or.kr/index.do">무인정보단말기UI플랫폼</a>
<button type="button">확인</button>
<input type="button" value="확인">
<input type="text">
<input type="checkbox" id="apple" name="사과">
<input type="radio">
<input type="file">
<input type="password">
<select name="pets" id="pet-select">
	<option value="dog">강아지</option>
	<option value="cat">고양이</option>
	...
</select>
<textarea id="story" name="story" rows="5" cols="30">
	내용을 작성해 주세요.
</textarea>
```
#### 2)	마우스와 키보드로 모두 조작할 수 있는 기능 제공
마우스와 같은 포인터로 접근 가능한 기능은 키보드로도 접근할 수 있어야 한다. 마우스를 올리면(hover) 보이거나 실행되는 버튼이나 메뉴는 키보드 초점을 받았을 때도 보이거나 실행되어야 한다. 참고로 툴팁은 보조 기술 지원을 고려하여 실행 요소와 설명문을 aria-describedby로 연결해 줄 수 있다.
```css
.share:hover + .share_tooltip,
.share:focus + .share_tooltip {
	display: block;
}
```
```html
<button class="share" aria-describedby="share-guide">공유</button>
<div id="share-guide" class="share__tooltip" role="tooltip">
	공유 기능으로 널리 퍼뜨려보세요! 모두에게 도움이 될거에요!
</div>
```
또한, 클릭 또는 터치 기능이 있다면 키보드로도 동등한 기능을 이용할 수 있어야 한다. 클릭 또는 터치 이벤트와 키보드 이벤트에 함께 대응하는 onclick과 같은 이벤트 핸들러를 사용하여 기능을 실행할 수 있도록 구현한다.
```html
<button type="button" onclick="uploadFile()">
	<img src="submit.png" alt="파일 업로드">
</button>
```
마우스를 가져가면 이미지 미리보기가 보이고 벗어나면 닫히는 사례를 키보드로도 동일하게 마우스 이벤트(mouseover, mouseout)와 키보드 초점 이벤트(focus, blur)로 동작하도록 구현할 수 있다.
```html
<input type="button" value="미리보기" id="previewButton">
<script>
	...
	const button = document.getElementById("previewButton");
	// 마우스 이벤트
	button.addEventListener("mouseover", () => process("true", "on"));
	button.addEventListener("mouseout", () => process("false", "off"));
	// 키보드 이벤트 (onfocus = hover 기능, onblur = mouseout 기능)
	button.addEventListener("focus", () => process("true", "on"));
	button.addEventListener("blur", () => process("false", "off"));
</script>
```

#### 3)	키보드로도 사용 가능한 사용자 정의 UI 컴포넌트
HTML의 기본 콤보박스(select), 입력폼(input) 등의 요소가 아닌, 사용자 정의 UI 컴포넌트(예: 콤보박스, 스위치 버튼, 슬라이더 등)도 키보드로 탐색하고 상호작용을 할 수 있도록 구현해야 한다.

이를 위해 각 컴포넌트는 Tab 키로 초점을 받을 수 있어야 하며, Enter 또는 Space 키를 눌렀을 때 특정 기능이 실행되도록 해야 한다.

키보드 초점을 받을 수 없는 요소에는 tabindex="0"을 부여하여 초점 이동을 가능하게 하고, keydown 이벤트를 추가하여 키보드 입력(키보드 인터페이스)으로도 동일한 기능이 실행될 수 있도록 구현해야 한다. 이때, 키보드 조작 방식은 HTML 기본 요소의 동작 방식을 참고하여 일관되게 적용해야 한다. 

추가로, ARIA 속성을 사용하여 UI의 역할과 상태 정보를 보조 기기에 전달될 수 있게 제공해야 한다.

<figure>
	![](/images/image123.png)
	<figcaption>이미지. 사용자 정의 스위치 UI 예시</figcaption>
</figure>

```html
<span
	role="switch"
	aria-checked="false"
	tabindex="0"
	onclick="handleClickSwitchBtn()"
	onkeydown="handleKeyDownSwitchBtn()">
	<span class="label"></span>
	<span class="switch"><span></span></span>
	<span class="status" aria-hidden="true">켬</span>
</span>
```

상호작용 가능한 HTML 기본 요소의 키보드 조작 방식의 일부는 다음과 같다.
- 라디오 버튼: name 값이 동일한 여러 개의 라디오 버튼은 위(↑), 아래(↓), 왼쪽(←), 오른쪽(→) 방향키로 탐색할 수 있다.
- 체크박스: Space 키를 눌러 선택/해제할 수 있다.
- 콤보박스(select 요소)
	- Enter 또는 Space 키를 눌러 목록을 펼칠 수 있다.
	- 펼쳐진 목록은 위(↑), 아래(↓) 방향키로 탐색할 수 있다.
	- Enter 키를 눌러 원하는 항목을 선택하면, 목록이 자동으로 닫힌다.
	- Esc 키를 눌러 펼쳐진 목록을 닫을 수 있다.
- 탭(Tab UI)
	- 탭 목록은 왼쪽(←), 오른쪽(→) 방향키로 탐색할 수 있다.
	- 탭 목록을 탐색하는 중 Tab 키를 누르면, 선택된 탭 패널로 이동한다.
</Card>

### 개선 필요 사례

#### 1)	키보드에는 대응하지 않는 CSS 효과
하위 메뉴가 마우스 :hover 상태에서만 펼쳐지는 경우, 키보드 사용자나 화면낭독 프로그램 사용자는 해당 하위 메뉴를 열 수 없다.
```css
.category-btn > a {
	display: block;
	margin-top: 73px;
	...
}
.category-btn:hover .category-layer {
	height: auto;
	display: block;
	...
}
```

##### 개선 방법
키보드로도 메뉴를 펼칠 수 있도록 `:focus` 스타일을 추가하거나, Javascript 이벤트 핸들러를 메뉴 실행 시 펼쳐지도록 구현한다.

#### 2)	초점을 받을 수 없어 키보드로 접근할 수 없는 버튼
img 요소에 직접 click 이벤트만 등록되어 있으면, 해당 요소는 키보드 초점을 받을 수 없으므로 Tab 키로 접근할 수도 없고, 키보드로 실행할 수도 없다.

<figure>
	![](/images/image124.png)
	<figcaption>이미지. img 요소로 구현된 버튼 사례</figcaption>
</figure>

```html
<div class="join__btns">
	<img src="…" alt="취소" class="join_bt" onclick="cancelJoin()">
	<img src="…" alt="가입" class=" btn-submit" onclick="join()">
</div>
```
##### 개선 방법
img 요소를 키보드 초점을 받을 수 있는 HTML 기본 요소인 button 요소로 구현한다.

어떠한 이유로 button 요소를 사용할 수 없다면, 다음과 같은 추가 작업이 필요하다. img 요소에 초점을 받을 수 있도록 tabindex="0"을 추가하고, 버튼임을 알리기 위해 role="button"을 설정한다. 그리고 keyup 이벤트를 추가하여 Enter 및 Space 키 입력 시 실행 가능하도록 설정한다. 

그러나 button 요소를 사용하면 별도의 키보드 이벤트를 추가할 필요 없이 접근성이 기본적으로 보장되므로, 개발 과정에서의 복잡성을 줄이고 유지보수가 쉬워진다.

```html
<img src="..." class="..." alt="가입" tabindex="0" role="button">
```

#### 3)	tabinedex=”-1”로 키보드 접근을 제한하였으나 화면낭독프로그램에서 접근되는 사례
tabindex="-1"은 키보드 Tab 키로 초점 이동이 불가능하지만, 화면낭독프로그램에서는 순차 탐색을 통해 해당 요소를 읽을 수 있어 접근이 가능하다. 따라서, 개발자가 의도적으로 키보드 초점에서 제외하려 했지만, 보조 기기 사용자에게는 노출되는 모순이 발생할 수 있다.

화면에 보이지 않게 숨긴 콘텐츠 내 tabindex="-1"이 적용된 버튼은 Tab 키로 이동할 수 없지만, 화면낭독프로그램 사용자는 탐색하고 실행이 가능하다. 결과적으로 키보드 사용자는 버튼을 조작할 수 없지만, 화면낭독프로그램 사용자는 접근 가능하여 일관되지 않은 사용자 경험이 발생한다.

```html
<div style="opacity:0">
	<p>이 항목은 중요한 정보를 포함하고 있습니다.</p>
	<button tabindex="-1">설정</button>
</div>
```

##### 개선 방법
화면에서 숨겨야 하는 경우, 보조 기술에서도 접근하지 못하도록 opacity:0 대신 display:none 속성을 적용한다.

부득이하게 display:none을 사용할 수 없는 경우, aria-hidden="true"를 적용하여 보조 기기의 접근을 막는다. 이때, aria-hidden 속성은 상호작용 요소에 직접 적용할 수 없으며, 대상을 감싸는 부모 요소에 적용해야 한다.

```html
<div style="opacity:0" aria-hidden="true">
	<p>이 항목은 중요한 정보를 포함하고 있습니다.</p>
	<button tabindex="-1">설정</button>
</div>
```

<Card title="참고. 이벤트 핸들러 선택에 대해" icon="open-book">
반응형 웹과 다양한 기기 환경이 보편화되면서, 웹페이지에서 여러 입력 방식(마우스, 키보드, 터치 등)에 대응해야 하는 필요성이 커졌다. 이벤트 핸들러는 사용자의 입력 방식에 따라 동일한 기능을 제공해야 하며, 입력 방식과 목적에 맞는 적절한 이벤트를 선택하는 것이 중요하다.

키보드 접근성을 위한 기본 이벤트로 `click`은 마우스 클릭뿐 아니라 키보드의 `Enter`나 `Space` 키로 요소가 활성화될 때를 포함하는 표준 이벤트이다. keydown은 특정 키를 눌렀을 때(주로 `Enter` 또는 `Space`) 발생하는 이벤트로, 키보드 전용 사용자들이 버튼 등과 같은 컨트롤을 사용할 때 필수적이다.

모든 마우스 전용 이벤트(`mousedown`, `mouseup`, `mouseover` 등)는 키보드 사용자에게 동일한 기능을 제공할 수 있도록 키보드 관련 이벤트(`keydown`, `keyup`, `focus`, `blur`)와 반드시 병행하여 제공한다.

###### 상황에 따른 이벤트 대응 예
| 상황  | 이벤트 대응 예 | 구현 설명 |
| --- | --- | --- |
| 커스텀 요소에 기본 버튼 동작 구현 시 | `click` + `keydown`(Enter/Space) | `role="button"` + `tabindex="0"` 필수 제공 |
| 터치가 끝나는 순간 이벤트 실행 | `touchend` | 상황에 따라 상호작용 요소에만 선택적으로 CSS `touch-action: manipulation`(예 더블탭 줌 방지) 추가 |
| hover 효과 | `mouseenter`/`mouseleave` + `focus`/`blur` | 키보드 사용자가 초점을 얻었을 때에도 효과를 동일하게 표시 |
</Card>

## 2.2 충분한 시간 제공

<Aside title="검사 항목">
시간제한이 있는 콘텐츠는 응답시간을 조절할 수 있어야 한다.
</Aside>

### 목적
시간제한이 적용된 콘텐츠를 제공할 경우, 사용자가 콘텐츠를 읽거나 양식을 제출하는 데 충분한 시간을 확보할 수 있도록 응답시간을 조절할 수 있는 기능을 제공해야 한다. 이를 통해 시각, 인지, 운동 등에 제약이 있는 사용자도 시간제한으로 인한 불이익 없이 웹 콘텐츠를 온전히 이용할 수 있도록 지원한다.

##	배경 및 필요성
예를 들어, 인증번호 입력과 같이 제한된 시간 내에 입력해야 하는 콘텐츠는 시각 장애, 인지 장애, 학습 장애 등을 가진 사용자에게 부담이 될 수 있다.

특히, 입력 시간이 오래 걸리거나 콘텐츠를 천천히 읽는 사용자는 제한된 시간 내 작업을 완료하기 어려워 오류가 발생하거나 정보를 놓칠 가능성이 높다.

화면낭독프로그램이나 키보드 사용자는 시간제한으로 인해 콘텐츠 변경이나 세션 종료 시 조작 시간이 더 걸리거나 주어진 시간 내 반응하기 어려울 수 있다. 이에, 사용자가 시간제한 기능을 해제하거나 연장할 수 있는 옵션을 제공하여, 사용자가 자신의 속도에 맞게 콘텐츠를 이용할 수 있도록 해야 한다.  

온라인 경매나 실시간 게임처럼 반응시간 조절이 원칙적으로 허용되지 않는 경우나, 세션 시간이 20시간 이상이면 예외로 본다. 이러한 경우에도, 사용자가 시간제한이 존재함을 미리 인지하고 종료 시점을 알 수 있도록 안내해야 한다.

##	기본 규칙
시간제한이 있는 콘텐츠를 제공해야 하는 경우, 다음 중 하나 이상의 기능을 제공해야 한다.
- 사용자가 시간제한 기능을 해제할 수 있는 설정을 제공해야 한다.
- 사용자가 남은 시간을 충분히 연장할 수 있는 기능을 제공해야 한다.
- 제한 시간을 연장할 수 있는 기능을 사용자가 쉽게 접근할 수 있도록 제공해야 한다. 제한 시간이 만료되기 최소 20초 전에 사용자에게 연장 가능 여부를 안내해야 한다. 사용자가 스페이스바 누르기 등 간단한 조작으로 시간제한을 10회 이상 연장하거나, 기본 제한 시간의 10배 이상을 추가로 확보할 수 있도록 해야 한다.
- 콘텐츠 시작 전에 시간제한이 적용됨을 미리 알리고, 제한 시간이 만료되면 종료됨을 안내해야 한다.
- 제한 시간이 조정 불가능한 경우(예: 온라인 경매, 실시간 게임, 20시간 이상 유지되는 세션)라도, 사용자에게 사전 안내를 통해 종료 시점을 명확히 알려야 한다.

### 개선 필요 사례

#### 1)	인증 시간제한에 연장 기능 미제공
제한된 인증번호 입력 시간에 연장 기능을 제공하지 않은 사례이다.

시간제한이 설정되어 있지만, 사용자가 이를 연장하거나 해제하는 방법이 제공되지 않았다. 이로 인해, 화면을 읽는 데 시간이 더 필요한 시각장애 사용자나, 키보드만으로 입력해야 하는 운동 장애 사용자는 인증번호를 입력하지 못해 절차를 완료할 수 없게 된다.

결과적으로 사용자는 서비스 접근이 차단되거나, 인증 절차를 반복해야 하는 불편을 겪을 수 있다.

##### 개선 방법
모든 사용자가 작업을 완료할 수 있도록 시간제한 해제 옵션을 제공하거나, 연장 기능을 추가해야 한다. 시간제한이 필수적인 경우, 사용자에게 이를 명확히 안내하고, 작업 실패 시 재시도 옵션을 제공하여 서비스 접근이 차단되지 않도록 해야 한다.

#### 2)	접속 시간제한에 연장 기능 미제공
작성 중이던 댓글을 등록하려 할 때, 접속 시간이 만료되어 자동으로 로그아웃되면서 댓글이 등록되지 않은 사례이다.

시간제한을 연장할 수 있는 기능이 없어 사용자는 제한 시간이 존재하는 사실을 인지하지 못한 채 작업을 진행하다가 예기치 않게 로그아웃되는 상황을 겪게 된다.  

또한, 시간 연장 기능이 제공되지 않으면 제한 시간 내 작업을 완료하지 못한 사용자는 작업을 처음부터 다시 시작해야 한다.

##### 개선 방법
접속 시간제한이 설정된 경우, 사용자에게 제한 시간이 있다는 사실을 명확히 안내해야 한다. 사용자가 페이지에 접속한 시점부터 남은 시간을 표시하거나, 일정 시간이 지나면 경고 메시지를 제공하여 제한 시간이 얼마나 남았는지 알릴 수 있어야 한다.

제한 시간이 만료되어도 작업 내용을 보존할 수 있도록, 자동 저장 기능을 추가하여 사용자가 작업을 처음부터 다시 시작하는 불편을 줄여야 한다.

#### 3)	자동 전환 페이지에 전환 시간 연장 기능 미제공
특정 페이지(URL)에 접속했을 때 자동으로 다른 페이지로 이동시키는 리다이렉트 페이지가 너무 빠르게 전환되어, 사용자가 안내 내용을 충분히 읽을 수 없는 사례이다. 

페이지 이동에 대한 주의 및 안내가 제공되었지만, 사용자가 페이지 내용을 충분히 인지하고 읽을 시간이 제공되지 않았다. 리다이렉트가 너무 빠르게 이루어지면, 사용자는 안내 정보를 확인하지 못하고 페이지 내용을 파악하기도 어려워진다.

<figure>
	![](/images/image154.png)

##### 개선 방법
자동으로 페이지가 전환되는 시간을 충분히 늘려, 사용자가 안내 내용을 읽고 이해할 수 있는 시간을 제공해야 한다.

또한, 페이지 전환을 자동으로 수행하지 않고, 사용자가 직접 전환을 선택할 수 있는 버튼이나 링크를 제공하는 방법도 고려해야 한다. 이 경우, 사용자에게 안내 메시지를 명확히 전달하고 전환 버튼의 위치와 기능을 쉽게 알 수 있도록 해야 한다.

## 2.3 광과민성 발작 예방

</Aside>

### 목적
[깜빡](../../glossary/blinking)이거나 [번쩍](../../glossary/flash)이는 콘텐츠는 광과민성 발작, 편두통, 현기증, 메스꺼움 등을 유발할 수 있으므로, 이를 방지하고 모든 사용자가 안전하게 콘텐츠를 이용할 수 있도록 하기 위함이다.

##	배경 및 필요성
광과민성 발작을 경험하는 사용자는 깜빡이는 콘텐츠의 주기, 크기, 변화 강도에 따라 발작이나 신체적 불편을 겪을 수 있다. 광과민성 발작은 단순한 불편함이 아니라, 심한 경우 의식 소실이나 전신 경련이 발생할 위험이 있어, 즉각적인 의학적 조치가 필요할 수 있다.

연구에 따르면 초당 20Hz(20회) 전후의 깜빡임이 광과민성 발작을 유발할 가능성이 가장 높으며, 전문가들은 초당 3회 이상의 깜빡임을 포함한 콘텐츠 사용을 제한할 것을 권장하고 있다.

전체 인구 중 약 100명 중 1명이 뇌전증을 앓고 있으며, 특히 어린이와 청소년이 더 높은 민감도를 보인다. 깜빡이는 콘텐츠는 발작뿐만 아니라 편두통, 현기증, 메스꺼움, 방향 감각 상실 등 신체적 불편을 유발할 수 있으며, 집중이 필요한 환경에서는 사용자의 주의를 방해할 수 있다.

이러한 요소를 고려할 때, 깜빡임과 번쩍임 사용 제한은 접근성을 넘어 사용자의 안전과 건강을 보호하기 위한 필수적인 요구사항이다.

##	기본 규칙
콘텐츠는 다음 사항 중 하나 이상을 충족해야 한다.
- 깜빡이거나 번쩍이는 시간을 3초 미만으로 제한하거나, 주기를 초당 3회 미만으로 제한한다.
- 10인치 이상의 화면에서 초당 3~50회로 깜빡이거나 번쩍이는 콘텐츠가 표시되는 경우, 해당 콘텐츠의 면적이 화면 전체의 10%를 초과하지 않도록 해야 한다. (단, 동영상과 같은 콘텐츠가 웹 페이지 일부에서 재생될 수 있고 전체 화면에서도 재생될 수 있는 경우, 전체 화면 기준으로 분석이 필요하다.)
- 깜빡이거나 번쩍이는 콘텐츠가 포함된 경우, 사용자에게 이를 사전에 안내하여 회피할 수 있도록 한다.
- 사전에 깜빡이거나 번쩍이는 콘텐츠를 정지할 수 있는 기능을 제공한다.

## 2.4 쉬운 내비게이션

<Aside title="검사 항목">
콘텐츠의 반복되는 영역은 건너뛸 수 있어야 한다.
</Aside>

### 목적
웹 페이지의 반복되는 영역(예: 내비게이션, 헤더, 메뉴 등)을 건너뛸 수 있는 기능을 제공하여, 키보드 사용자와 화면낭독프로그램 사용자가 주요 콘텐츠로 빠르게 이동할 수 있도록 한다. 이를 통해 불필요한 반복 탐색을 줄이고, 보다 효율적으로 콘텐츠를 이용할 수 있다.

##	배경 및 필요성
대부분의 웹 페이지에는 내비게이션 링크, 헤더, 광고 등 공통 요소가 반복적으로 포함되어 있다. 키보드 사용자와 보조 기술 사용자(예: 화면낭독프로그램 사용)는 이러한 반복되는 영역을 일일이 탐색해야 하므로, 주요 콘텐츠에 도달하기까지 많은 시간이 소요될 수 있다.

특히, 탐색해야 하는 내비게이션 링크가 많은 웹사이트에서는 키보드 사용자가 수십 번의 Tab 키를 눌러야 본문에 도달하는 경우도 있다. 이러한 불편을 줄이기 위해 웹 페이지의 주요 본문으로 빠르게 이동할 수 있는 ‘건너뛰기 링크(skip link)’를 제공한다. 

‘건너뛰기 링크’는 페이지의 최상단, body 요소 시작 직후에 배치하는 것이 필요하다.  키보드 접근성을 보장하기 위해 Tab 키로 접근할 수 있어야 한다. 건너뛰기 링크는 항상 시각적으로 표시하거나, 키보드 초점을 받았을 때만 표시할 수 있지만, 반드시 기능적으로 작동해야 한다. 단순히 링크만 제공하고 실제 작동하지 않는다면 접근성을 준수했다고 볼 수 없다.

예외적으로, 페이지에 반복되는 영역(예: 헤더, 내비게이션 바 등)이 없는 경우 반복 영역 건너뛰기 링크는 생략할 수 있다.

##	기본 규칙
- 웹 페이지의 가장 앞부분에 키보드로 핵심 콘텐츠 영역으로 바로 이동할 수 있는 ‘건너뛰기 링크’를 제공해야 한다.
- 여러 개의 건너뛰기 링크를 제공할 경우, 핵심 콘텐츠(본문)로 이동하는 링크가 가장 먼저 나타나도록 순서를 정한다.
- 건너뛰기 링크는 키보드 사용자와 보조 기술 사용자가 쉽게 인식할 수 있도록 눈에 띄게 구현해야 한다.

### 적용 사례
#### 1)	반복 영역 건너뛰기 링크 제공
웹 페이지 상단에 ‘본문 바로가기’ 링크를 배치하여, 키보드 탐색 시 사용자가 반복되는 내비게이션 영역을 건너뛰고 바로 본문으로 이동할 수 있도록 하였다. 건너뛰기 링크는 기본적으로 숨겨져 있지만, 키보드 초점을 받을 때 화면에 나타나도록 구현하여 사용자가 쉽게 인식할 수 있도록 하였다.

### 개선 필요 사례
#### 1)	반복 영역 건너뛰기 미제공
반복되는 상단(`header`) 영역이 있는 페이지에서 ‘본문 바로가기’ 링크가 제공되지 않으면, 키보드나 보조 기술 사용자가 매번 반복되는 영역을 탐색해야 한다.

<figure>
	![](/images/image166.png)
	<figcaption>이미지. 반복 영역 건너뛰기 링크가 제공되지 않은 사례</figcaption>
</figure>

##### 개선 방법
페이지 최상단에 '본문 바로가기' 링크를 추가하여, 사용자가 핵심 콘텐츠로 바로 접근할 수 있도록 한다.

#### 2)	바로가기 링크 남용
키보드 접근 시 노출되는 바로가기 링크 모음이다. 반복 영역 건너뛰기가 아닌 각 콘텐츠 블록으로 이동하는 다수의 바로가기 링크를 추가로 제공하여, 오히려 키보드 사용자의 반복 이동 콘텐츠가 늘어나 탐색 단계가 증가한 사례이다.

이 경우, 본래의 ‘본문 바로가기’ 기능 역할을 하지 못하며, 화면낭독프로그램 사용자와 키보드 사용자가 페이지 구조를 파악하고 탐색하는 데 시간이 오래 걸리게 된다.

<figure>
	![](/images/image167.png)
	<figcaption>이미지. 키보드 접근 시 활성화되는 다수의 바로가기 링크 제공 사례</figcaption>
</figure>

##### 개선 방법
반복 영역 건너뛰기 링크는 반드시 본문 시작 영역으로 이동하도록 설정하고 페이지 상단에 배치해야 한다. 또한, 페이지 내 다른 콘텐츠 블럭 이동 링크와 혼동되지 않도록 명확히 구분하여 제공해야 한다.

필요할 경우, 별도의 내비게이션 체계를 통해 역할을 분리함으로써 사용자가 효율적으로 탐색할 수 있도록 한다.

#### 3)	잘못된 링크 연결
건너뛰기 링크에 연결된 id="content" 영역이 웹 페이지 내 존재하지 않아, 바로가기 기능이 작동하지 않는 사례이다. 이로 인해 키보드 사용자와 화면낭독프로그램 사용자는 반복 영역을 모두 탐색해야 하는 불편을 겪는다.
```html
<body>
    <!-- 본문 바로가기 -->
    <div id="skip" class="page-skip">
        <a href="#content" class="skip-item">본문 바로가기</a>
    </div>
    <!-- 메뉴 -->
    <div class="page-head" id="head">
        ...
    </div>
    <!-- 본문 영역 -->
    <div class="page-body">
        <!-- event container -->
        <div class="event-container">...</div>
    </div>
</body>
```
##### 개선 방법
건너뛰기 링크는 반드시 본문 시작 영역(예: id="content")에 id 값을 지정하고, 링크 href 속성이 해당 영역의 id와 올바르게 연결되도록 수정해야 한다.

#### 4)	본문 아닌 다른 영역으로 연결된 건너뛰기 링크
반복 영역 건너뛰기 링크가 본문 시작 영역 대신 본문 중간에 추천 상품 영역에 연결된 사례이다. 이에 따라 키보드 사용자와 화면낭독프로그램 사용자는 본문 시작 부분부터 탐색하지 못하고, 중간에 있는 추천 상품 영역 이전의 콘텐츠를 인식하기 어렵게 된다.

```html
<body class="body-main">
	<a href="#best_goods">본문바로가기</a>
	<div id="wrap">
		<div id="header_wrap" class="headerFix">...</div>
			<div id="container" style="padding-top: 165px;">
				<!-- 이동해야 하는 본문 시작 위치(O) -->
				<div id="contents">
					<!-- 본문 시작 -->
					<div class="sub_content">
						<div class="main_visual">...</div>
						</div>
						<div class="main_content">
							<div class="main_banner_list">...</div>
						</div>
					</div>
					<!-- 본문 중간으로 잘못 연결된 위치(X) -->
					<div id="best_goods" class="main_goods_cont">
						<div class="goods_list">
							<div class="goods_list_tit">
								<h3>추천상품</h3>
							</div>
				...
			</div>
		</div>
	</div>
</body>
```

##### 개선 방법
반복 영역 건너뛰기 링크는 본문 시작 지점과 연결되도록, 본문 시작 지점에 적절한 식별자를 설정하고 정확히 연결해야 한다.

위 사례는, `<a href="#contents">본문 바로가기</a>`로 제공해야 한다.

#### 5)	화면만 스크롤 되는 반복 영역 건너뛰기 링크
건너뛰기 링크를 클릭했을 때 화면만 본문 영역으로 스크롤 되어 보여주고, 키보드 초점은 그대로 건너뛰기 링크에 유지되는 사례이다.

이 경우, 키보드 사용자와 화면낭독프로그램 사용자는 실제로 본문 영역으로 초점이 이동하지 않아 반복 영역을 계속 탐색해야 한다.

```javascript
document.addEventListener("DOMContentLoaded", function () {
	document.querySelectorAll(".skip-link").forEach(function (link){
		link.addEventListener("click", function (event) {
			event.preventDefault();
			const targetId = this.getAttribute("href").substring(1);
			document.getElementById(targetId).scrollIntoView();
		});
	});
});
```
##### 개선 방법
반복 영역 건너뛰기 링크를 클릭하면 화면 스크롤뿐만 아니라 키보드 초점도 본문 시작 영역으로 이동하도록 구현해야 한다. 이렇게 하면 모든 사용자가 바로 주요 콘텐츠를 탐색할 수 있게 된다.

## 2.5 입력 방식

<Aside title="검사 항목">
다중 포인터 또는 경로기반 동작을 통한 입력은 단일 포인터 입력으로도 조작할 수 있어야 한다.
</Aside>

### 목적
터치를 사용할 수 있는 모바일 기기 등에서 두 개 이상의 손가락을 동시에 사용하는 다중 포인터 동작, 정밀한 경로나 모양을 그려야 하는 경로 기반 동작, 끌기와 놓기(<span lang="en">drag & drop</span>) 동작 등은 일부 사용자에게 어렵거나 수행하지 못할 수 있다. 이러한 사용자도 한 손가락과 최소한의 동작만으로 원활하게 웹 콘텐츠와 상호작용을 할 수 있도록 하기 위함이다.

따라서, 다양한 입력 장치를 사용하는 모든 사용자가 웹 콘텐츠와 원활하게 상호작용을 할 수 있도록, 다중 포인터 또는 경로 기반 동작으로 실행되는 기능은 단일 포인터 입력만으로도 조작할 수 있어야 한다.

##	배경 및 필요성
모바일 기기에서 스와이프 제스처에 의해 기능이 수행되는 경우, 사용자는 화면을 터치한 채 한 방향으로 쓸어서 포인터를 이동시켜야 한다. 그러나 일부 사용자는 일정한 압력을 유지하며 동일한 방향으로 제스처를 취하기 어렵다.

마찬가지로, 두 손가락을 사용하는 동작(예: 두 손가락으로 쓸어내리는 동작, 핀치 줌 등)은 일부 장애가 있거나 한 손가락밖에 사용할 수 없는 사용자는 수행할 수 없다.

또한 노트북의 터치패드를 사용할 때, 사용자가 파일을 끌어다 놓기 위해 한 손가락은 누름 동작을 유지한 채 다른 손가락을 사용해 폴더를 이동해야 하는 경우가 있다. 이러한 동작은 손가락 움직임에 제약이 있는 사용자나 키보드 사용자가 수행하기 어렵다.

따라서, 다중 포인터나 경로 기반 동작으로 실행되는 기능은 단일 포인터 입력만으로도 동일한 기능을 수행할 수 있는 대체 사용자 인터페이스를 제공해야 한다.

다음과 같은 경우에는 예외로 한다.

서명 같은 자유로운 그리기 기능, 피아노 건반처럼 동시에 여러 개의 입력이 필수적으로 다중 포인터가 필수적인 기능이거나 단순화할 수 없는 기능, 화면 가장자리에서 안쪽으로 스와이프하여 메뉴를 여는 등 브라우저나 운영체제에서 기본으로 제공되는 표준 제스처가 해당한다.

##	기본 규칙
- 다중 포인터가 필요한 동작은 단일 포인터 입력으로도 수행할 수 있도록 한다.
- 경로 기반 동작이나 특정 모양을 그리는 대신, 단일 포인터 입력만으로 기능을 실행할 수 있는 대체 인터페이스를 제공한다.

단일 포인터 입력으로도 조작할 수 있는 대체 수단 제공의 구체적인 예로 다음과 같은 것을 고려할 수 있다.

- 두 손가락 핀치 줌은 한 손가락으로 조작 가능한 ‘+’, ‘-’ 버튼을 제공한다.
- 방향을 돌리기 위한 회전 제스처는 회전 버튼으로 제공한다.
- 파일이나 항목을 끌기와 놓기(drag & drop)는 ‘선택’ 후 ‘이동’ 버튼을 제공한다.
- 경로 기반 동작이나 특정 모양을 그리는 대신 해당 기능을 실행하는 버튼을 제공한다.

이러한 대체 수단은 사용자가 쉽게 발견할 수 있어야 하며, 기존 제스처나 끌기(Dragging) 동작으로 실행되는 기능과 동등한 기능을 제공해야 한다.

# 3. 이해의 용이성

이해의 용이성이란 콘텐츠와 인터페이스가 명확하고 일관된 방식으로 제공되어, 사용자가 혼란을 겪지 않고 콘텐츠를 이용할 수 있도록 돕는 것이다. 언어적 장애가 있거나, 콘텐츠를 빠르게 이해해야 하는 상황에서도 정보를 쉽고 정확하게 이해할 수 있도록 제작해야 한다.

다음과 같은 상황에서 도움이 된다.
- "공지사항 더보기"와 "이벤트 더보기" 버튼처럼 명확히 구분하면, "더보기" 같은 막연한 표현만 있을 때보다 사용자가 목적한 메뉴를 쉽게 찾을 수 있다.
- 회원 정보 수정 후 "저장되었습니다" 같은 알림을 시각적으로만 제공하는 것이 아니라, 화면낭독프로그램에서도 확인할 수 있도록 하면, 시각 장애가 있는 사용자도 결과를 파악할 수 있다.
- 오류 메시지를 단순히 "에러가 발생했습니다"로 표시하는 대신, "이메일 형식이 올바르지 않습니다. 예) `example@domain.com`"처럼 안내하면, 사용자가 문제 해결 과정을 쉽게 이해할 수 있다.

이해의 용이성이 개선되면, 사용자는 처음 방문하더라도 사이트의 흐름을 빠르게 파악할 수 있으며, 장애 유무와 관계없이 원활하게 탐색할 수 있다.

## 3.1 가독성

<Aside title="검사 항목">
사용하는 언어를 명시해야 한다.
</Aside>

### 목적
웹 페이지의 주 사용 언어를 명확히 지정하여, 보조 기술(화면낭독프로그램 등)과 사용자 에이전트가 텍스트를 올바르게 해석하고, 사용자에게 정확한 발음과 번역, 문자 표시를 제공할 수 있도록 보장한다.

##	배경 및 필요성
웹 페이지의 언어가 명확히 지정되어 있으면 화면낭독프로그램은 기본 언어에 맞는 발음 규칙을 적용하고, 시각적 브라우저는 해당 언어에 맞는 문자를 올바르게 표시할 수 있다. 예로, 한국어 문서에 언어 코드가 en으로 적용되었을 때 화면낭독프로그램은 영어 음성 엔진을 통해 영어 억양으로 한국어를 읽어 어색함이 느껴진다.

따라서 웹 페이지 전체에서 주로 사용되는 언어를 HTML의 lang 속성을 통해 명확히 지정해야 한다. 추가로 기본 언어와 다른 언어로 작성된 콘텐츠가 포함된 경우, 해당 부분에 별도로 언어를 지정해주면, 이를 통해 보조 기술은 언어 변화를 인식하여 처리할 수 있다.

##	기본 규칙
- 웹 페이지 전체에서 주로 사용되는 언어는 HTML의 최상위 html 요소의 lang 속성(예: `<html lang="ko">`) 으로 명확히 지정해야 한다.
- HTML 요소의 `lang` 속성에는 반드시 유효한 값을 입력해야 하며, 이는 국제 표준인 BCP 47 규칙을 따라 작성한다. 예를 들어, 한국어는 ko, 영어는 en, 번체 중국어는 zh-Hant를 사용하며, ISO 639.2의 3글자 코드는 사용하지 않는다.

<Card title="참고. BCP 47에 대해" icon="open-book">
BCP 47은 웹 페이지의 기본 언어를 지정할 때 사용하는 국제 표준이다.

이 규격을 올바르게 적용하면, 브라우저와 보조 기술(예: 화면낭독프로그램)이 해당 페이지의 언어를 인식하여 적절한 발음, 문자 표시, 번역 등을 지원할 수 있다.

이 규격은 주로 ISO 639의 2글자 코드(예: ko는 한국어, en은 영어)를 사용하며, 필요시 지역 코드(예: en-US는 미국 영어, en-GB는 영국 영어)도 추가할 수 있다. 다만, 특별한 필요가 없다면 지역 코드는 생략하고 기본 언어 코드만 사용하는 것이 권장된다. 예를 들어, 일본어는 ja-JP 대신 간단히 ja를 사용하며, 한국어도 ko-KR보다 ko로 충분하다.

**표. 언어코드 조합 예**

| **코드** | **언어** | **하위 요소** |
| --- | --- | --- |
| en  | 영어  | 언어  |
| mas | 마사이어 | 언어  |
| fr-CA | 캐나다에서 사용되는 프랑스어 | 언어+지역 |
| es-419 | 라틴 아메리카에서 사용되는 스페인어 | 언어+지역 |
| zh-Hans | 간체로 작성된 중국어 | 언어+스크립트 |
</Card>

### 적용 사례

#### 1)	lang 속성으로 주 사용 언어 정보 명시
웹 페이지의 주 언어를 확인 후, lang 속성을 통해 해당 언어를 명시한다.\

```html
<!DOCTYPE html>
<html lang="ko">
<head>
  ...
</head>
<body>
    <div>오늘의 날씨는 최저 -6도이며 최고 -1도 입니다. 추위에 유의하시기 바랍니다</div>
```

#### 2)	(권고) 주 언어와 다른 언어 사용 콘텐츠의 언어 명시
웹 페이지에 주 사용 언어와 다른 언어로 작성된 콘텐츠가 포함될 경우, 해당 콘텐츠의 언어를 지정해야 한다.

이를 통해 보조 기술(예: 화면낭독프로그램)이 언어 전환을 인식하여, 사용자가 콘텐츠를 더 자연스럽게 이해할 수 있다. 언어를 명시하지 않으면 음성 정보만 제공되었을 때 발음 차이로 의미가 다르게 해석되는 상황도 발생할 수 있다.

아래와 같이, 주 사용 언어가 한국어인 페이지에서 중국어 단어나 구문이 사용 시, 해당 부분에 lang 속성을 부여해 중국어임을 명시해 준다. 간체 중국어는 zh를, 번체 중국어는 zh-Hant를 사용해 언어를 지정할 수 있다. 이렇게 하면 화면낭독프로그램는 한국어에서 중국어로 전환되는 부분을 인식하고, 적절한 발음으로 출력할 수 있다.

```html
<html lang="ko">
<head>
	...
</head>
<body>
    <div>
        <p>중국어로 "안녕하세요"는 <span lang="zh">你好</span> 입니다.</p>
        <p>프랑스어로 "안녕하세요"는 <span lang="fr">Bonjour</span> 입니다.</p>
    </div>
    ...
```

### 개선 필요 사례
#### 1)	언어코드 미제공
기본 언어를 명시하지 않으면, 화면낭독프로그램 등 보조 기술이 페이지의 언어를 올바르게 인식하지 못하여 잘못된 발음이나 번역 오류가 발생할 수 있다. 특히, 다국어 환경에서 기본 언어를 설정이 중요하다. 

```html
<html>
<head>
	<title>ㅇㅇ기업</title>
	...
```

##### 개선 방법
페이지의 기본 언어를 명확히 하기 위해 `<html lang="ko">`와 같이 BCP 47 표준에 따른 올바른 언어 코드를 지정한다.

```html
<html lang="ko">
```

#### 2)	페이지의 주 언어와 다른 언어 명시
다음은 주 사용 언어와 다른 언어를 기본 언어로 잘못 명시한 사례이다.

화면낭독프로그램은 lang 속성을 참고하여 해당 언어에 맞는 발음 엔진을 사용한다. 따라서 한국어 콘텐츠지만 영어(lang="en")로 설정된 경우, 한국어 발음을 지원하지 않는 음성 엔진이 사용되거나, 부정확한 발음으로 음성이 출력될 수 있다.

```html
<!DOCTYPE html>
<html lang="en">
<head>
	...
</head>
<body>
<main>
	<div>
		<img src="..." alt="많이 살수록 더 저렴하죠 대용량 특가">
		<span>매일 필요한 생수, 세제, 화장지, 샴푸는</span>
		<span>미리 쟁여두면 든든하죠.</span>
	</div>
</main>
...
```

##### 개선 방법
페이지의 주 사용 언어를 올바르게 설정해야 한다. 페이지의 주 언어가 한국어라면 `<html lang="ko">`와 같이 유효한 언어 요소를 이용하여 lang 속성을 설정해야 한다.

```html
<html lang="ko">
```

#### 3)	올바르지 않은 언어 코드 제공
다음은 웹 페이지에서 사용하는 언어를 잘못 설정한 사례이다.

한국어로 작성된 웹페이지에 ko_KR이라는 잘못된 코드를 사용하였다. 이 코드는 표준에 맞지 않아 화면낭독프로그램이 언어를 올바르게 인식하지 못한다.

```html
<!DOCTYPE html>
<html lang="ko_KR">
<head>
  ...
</head>
<body>
	<div>오늘의 날씨는 최저 -6도이며 최고 -1도 입니다. 추위에 유의하시기 바랍니다</div>
	...
```

##### 개선 방법
웹 페이지에서 사용하는 언어를 설정할 때는 국제 표준인 BCP 47에 따라 올바른 언어 요소를 사용해야 한다. 한국어로 작성된 웹 페이지라면 `lang` 속성값을 ko로 설정해야 한다. ko-KR처럼 지역 정보를 포함한 요소도 사용할 수 있지만, 대부분은 ko와 같은 간단한 언어 코드만으로 충분하다.

```html
<html lang="ko">
```

#### 4)	(권고) 주 언어 외 콘텐츠 언어 속성 누락
다음은 주 사용 언어와 다른 언어로 제공된 일부 콘텐츠에 언어가 명시되지 않은 사례이다.

한국어와 일본어가 혼용된 콘텐츠에서 일본어에 대한 언어 속성이 지정되지 않으면, 화면낭독프로그램이 일본어를 올바르게 인식하지 못해 부정확한 발음으로 음성 출력을 하게 될 수 있다. 이에 따라 콘텐츠를 음성으로 듣는 사용자가 일본어 부분을 제대로 이해하지 못하는 문제가 발생할 수 있다.

```html
<!DOCTYPE html>
<html lang="ko">
<head>
  ...
</head>
<body>
  ...
	<p>
		おはようございます로 아침을 시작해 おやすみなさい를 마지막으로 잠자리에 들 때까지 자연스럽게 일본어가 나올 수 있도록
	</p>
	<button class="btn_join">학습 시작하기</button>
...
```

##### 개선 방법
주 사용 언어와 다른 언어로 제공된 콘텐츠는 해당 언어를 명확히 명시해야 한다. 예를 들어, HTML의 `lang` 속성을 사용하여 일본어로 작성된 부분에 `lang="ja"`를 추가하면 된다.

이렇게 하면 화면낭독프로그램이 해당 부분이 일본어임을 인식하고, 그 언어에 맞는 발음 규칙에 따라 음성을 출력할 수 있다. 이를 통해 일본어 콘텐츠를 이용하는 사용자도 정확한 정보를 얻을 수 있다.

```html
<!DOCTYPE html>
<html lang="ko">
<head>
  ...
</head>
<body>
  ...
	<p>
		<span lang="ja">おはようございます</span>로 아침을 시작해
		<span lang="ja">おやすみなさい</span>를 마지막으로 잠자리에 들 때까지 자연스럽게 일본어가 나올 수 있도록
	</p>
	<button class="btn_join">학습 시작하기</button>
...
```

## 3.2 예측 가능성

<Aside title="검사 항목">
사용자가 의도하지 않은 기능(새 창, 초점에 의한 맥락 변화 등)은 실행되지 않아야 한다.
</Aside>

### 목적
사용자가 버튼 클릭이나 키보드 입력과 같은 명확한 조작을 통해 기능을 실행할 때만 콘텐츠가 변경되도록 하여, 예측할 수 있는 상호작용을 보장하는 것이 목적이다. 특히, 장애가 있는 사용자는 예상치 못한 새 창 열림이나 체크박스 선택 후 갑작스러운 페이지 이동, 초점 이동에 따른 맥락 변화로 인해 큰 혼란을 겪을 수 있으므로 이를 방지해야 한다.

##	배경 및 필요성
사용자는 일반적으로 버튼을 클릭하거나 키보드의 Enter 키를 눌렀을 때 기능이 실행될 것으로 예상한다. 그러나, 초점을 받거나 체크박스 및 콤보박스에서 항목을 선택하는 것만으로 기능이 실행될 것이라고는 예상하지 않는다.

화면낭독프로그램을 사용하는 시각장애인은 현재 페이지의 흐름을 파악하는 중에 자동으로 새 창이 열리거나 초점이 이동하면 원래의 흐름을 놓칠 수 있다. 이는 사용자가 예상하지 못한 상황에서 맥락을 이해하기 어렵게 만든다. 운동 장애가 있는 사용자는 의도치 않은 새 창 이동으로 인해 원래 창으로 돌아오는 것이 어려울 수 있다. 이는 특히 키보드나 특수 장치를 사용하는 사용자에게 큰 어려움이 될 수 있다.

콤보박스에서 항목을 변경했을 때 별도의 확인 없이 자동으로 페이지가 이동하면, 키보드 사용자나 화면낭독프로그램 사용자는 예기치 않은 맥락 변화로 인해 현재 위치를 잃을 수 있다. 특정 인지적 제한이 있는 사용자는 갑작스러운 맥락 변화가 발생하면 정보 탐색에 어려움을 겪을 수 있다.

마우스가 올라가거나(hover) 키보드 초점을 받았을 때 버튼 색상이 변하거나 그림자가 생기는 등 시각적 효과가 발생하는 것은 단순한 피드백으로 간주한다. 이는 사용자에게 현재 상태를 알려주는 역할을 하며, 기능 실행에는 영향을 미치지 않는다.

하지만 마우스를 올리거나 초점을 받는 것만으로 새 창이 열리거나, 페이지가 변경되는 등 맥락의 변화가 발생하면 사용자가 의도하지 않은 기능 실행이라 오류로 본다.

##	기본 규칙
- 모든 콘트롤이나 입력 기능은 사용자의 명시적 조작(클릭, Enter 키 입력 등)으로만 실행되어야 한다.
- 새 창이나 팝업은 사용자가 요청하거나 조작한 경우에만 열려야 한다.
- 초점이 이동하거나 포인터가 올려진 상태에서는 단순 시각적 피드백(예: 색상 반전, 테두리 변화)만 허용되며, 기능이 자동으로 실행되지 않아야 한다.

### 적용 사례

#### 1)	사용자 의도에 의한 명확한 실행
관련 사이트로 이동하는 기능을 콤보박스로 구현할 때, 사용자는 옵션 선택 목록에서 이동할 사이트를 선택한 후 별도의 ‘이동’ 버튼을 실행해야 한다. 이러한 방식은 키보드 사용자가 방향키로 목록을 탐색하는 동안 원치 않는 사이트로 이동하는 문제를 방지할 수 있다.

#### 2) 새 창 열림 정보 사전 제공
새 창에서 콘텐츠가 열릴 경우, 사용자는 기존 창과 새 창 사이의 관계 및 새 창의 콘텐츠 연관성을 쉽게 파악하기 어려울 수 있다. 특히, 화면낭독프로그램 사용자는 새 창 열림을 사전에 알지 못하면 웹 탐색 흐름이 끊어질 수 있다.

링크 요소에 target="_blank" 속성만 추가한 경우, 대부분의 화면낭독프로그램이 이를 사용자에게 전달하지 못한다. 사용자가 콘텐츠 맥락의 변화를 예측할 수 있도록 별도로 새 창 열림 정보를 제공해야 한다.

이를 위해 화면에는 보이지 않지만, 화면낭독프로그램이 인식할 수 있는 숨김 텍스트를 추가하여 제공할 수 있다.

또한, 모든 사용자가 동등한 정보를 인식할 수 있도록 링크 옆에 새 창 아이콘을 추가하거나 '새 창 열림'과 같은 텍스트를 제공하여 시각적으로 구분되는 정보를 함께 제공하는 방법도 있다.

### 개선 필요 사례
#### 1)	새 창 열림 정보 미제공
링크에 `target="_blank"` 속성만 적용된 경우로, 새 창이 열림을 알리는 시각적 단서나 텍스트 안내가 없어, 화면낭독프로그램 사용자 등 대부분의 사용자는 이를 인지하지 못한다.

이에 따라 사용자는 예기치 않게 새 창이 열리고, 원래 창으로 돌아가기 위해 추가 조작이 필요하다.

<figure>
	![](/images/image223.png)
	<figcaption>이미지. 시각적인 단서 없이 새 창으로 열리는 링크를 사용한 예시</figcaption>
</figure>

```html
<ul>
  <li><a href="..." target="_blank">접근성 센터</a></li>
  <li><a href="..." target="_blank">접근성 컨설팅</a></li>
	...
</ul>
```

##### 개선 방법
화면낭독프로그램에서 새 창이 열리는 것을 인식할 수 있도록 링크 옆에 숨김 텍스트(예: (새 창 열림))를 추가한다.

```html
<a href="..." target="_blank">접근성 센터<span class="visually-hidden">(새 창 열림)</span></a>
```
참고로, `title=”새창열림”`은 화면낭독프로그램 설정이나 조작 방식에 따라 사용자에게 전달되지 않을 수 있으므로 중요한 정보가 누락되지 않도록 `title` 속성 사용은 지양한다. 

추가로 새 창으로 열리는 링크에 시각적인 아이콘을 함께 제공하면, 모든 사용자도 쉽게 인식할 수 있도록 할 수 있다.

#### 2)	사용자가 의도하지 않은 기능 실행
콤보박스(change 이벤트)의 값 변경 시 자동으로 페이지를 이동시키면, 키보드 사용자와 화면낭독프로그램 사용자가 방향키로 옵션을 탐색하는 동안 의도치 않게 페이지가 전환된다.

이로 인해 다른 옵션을 선택하기 어려워진다.

<figure>
	![](/images/image224.png)
	<figcaption>이미지. change 이벤트에 자동으로 페이지가 이동하는 콤보박스 예시</figcaption>
</figure>

```javascript {3}
// family-site 셀렉트 박스에서 사용자가 선택을 바꾸면, 해당 주소로 이동하는 예시 코드
document.getElementById('family-site').addEventListener(
	'change',
	function(event) {
		var selectedUrl = event.target.value;
		// 이동할 URL 값이 존재한다면 해당 URL로 이동
		if (selectedUrl) {
			window.location.href = selectedUrl;
		}
	}
);
```

##### 개선 방법
콤보박스에서 옵션을 탐색할 때는 단순히 포커스 이동만 일어나고, 페이지 이동은 사용자가 별도의 '이동' 버튼을 클릭한 후에 실행되도록 구현한다.

#### 3)	페이지 진입 시 새 창 팝업 열림
페이지에 진입하면 게시판 관련 공지가 새 창 팝업으로 제공되는 사례이다.

팝업 창이 자동으로 열릴 경우, 팝업 창 실행이 사용자 동의 없이 이루어졌기 때문에 화면낭독프로그램 사용자는 자신이 다른 페이지로 이동했다고 혼동할 수 있고 따라서 현재 위치를 명확히 이해하지 못할 수 있다.

	<figcaption>이미지. 페이지 진입 시 팝업 창이 자동으로 열리는 사례</figcaption>
</figure>

```javascript
window.addEventListener("load", function () {
	window.open("notice.html", "_blank", "width=300,height=400");
```

##### 개선 방법
팝업 창이 자동으로 실행되지 않도록 설정하고, 사용자가 명확히 이해하고 선택할 수 있는 방식으로 제공한다.

공지사항은 팝업 창이 아닌 페이지 내 특정 영역에 제공하거나, '공지사항 보기'와 같은 버튼을 통해 사용자가 직접 팝업을 열 수 있도록 해야 한다.

## 3.3 입력 도움

<Aside title="검사 항목">
입력 오류를 정정할 수 있는 방법을 제공해야 한다.
</Aside>

### 목적
사용자가 입력 오류를 쉽게 인지하고, 어떤 부분이 잘못되었는지 명확한 오류 안내와 정정 방법을 제공하여 오류를 파악하고 수정할 수 있도록 돕기 위한 것이다. 오류 정정 기능을 제공함으로써, 사용자는 입력 과정에서 발생할 수 있는 실수를 빠르게 수정하고 원활하게 서비스를 이용할 수 있도록 지원한다.

##	배경 및 필요성
입력 오류가 발생했을 때, 사용자는 무엇이 잘못되었는지 빠르게 파악하고 이를 수정할 수 있어야 한다. 장애가 있는 사용자는 입력 양식을 작성하는 데 비장애인보다 더 많은 시간이 소요될 수 있고, 오류 안내가 부족할 경우 서비스 이용을 포기할 가능성도 있다.

오류 안내는 화면낭독프로그램과 같은 보조 기술이나 사용자 에이전트(웹 브라우저 등)가 오류 정보를 식별하여 전달할 수 있도록 텍스트 형태로 제공되어야 한다. 또한, 사용자가 오류를 쉽게 인지하고 수정할 수 있도록 텍스트 외에도 이미지, 색상, 기호 등 다양한 시각적 요소를 보조 수단으로 활용할 수 있다. 단, 시각적 정보만으로 오류를 전달해서는 안 되며, 모든 사용자가 접근할 수 있는 방식으로 정보를 제공해야 한다.

더불어, 인지 능력에 제한이 있거나 시각적 정보를 활용하기 어려운 사용자도 오류를 수정할 수 있도록 지원해야 하며, 오류 메시지에는 구체적인 정정 방법과 오류가 발생한 위치 안내가 포함되어야 한다. 예를 들어, '이름 입력란은 공백으로 둘 수 없습니다. 이름을 입력해 주세요.'와 같은 명확한 지침을 제공하고, 이름 입력란에 초점이 자동으로 이동하도록 하여 사용자가 문제를 확인하고 해결할 수 있도록 도울 수 있다.

##	기본 규칙
- 입력 오류 발생 시, 사용자가 즉시 오류를 인지할 수 있도록 명확한 오류 메시지를 제공해야 한다. 오류 안내는 전체 입력 양식에 걸쳐 일관되게 적용되어야 한다.<br/>예) 생년월일 입력란에서 '생년월일은 8자리로 입력해 주세요.'와 같이 구체적인 안내를 제공한다.
- 오류 메시지는 사용자가 오류를 어떻게 정정할 수 있는지 명확한 지침을 포함해야 한다.<br/>예) '오류가 있습니다.'라는 메시지 대신 '비밀번호는 최소 8자 이상이며, 숫자와 특수문자를 포함해야 합니다.'와 같이 구체적인 요구사항을 명시한다.
- 오류 정보는 색상만으로 제공하지 않으며, 시각적 요소(색상, 기호, 이미지 등)는 오류 정보를 보완하는 수단으로 활용할 수 있다.<br/>예) 필수 입력 항목을 색상(빨간색 테두리)만으로 표시하지 않아야 한다.
- 오류가 발생한 입력 요소에 초점을 이동시켜, 사용자가 오류 위치를 쉽게 확인하고 수정할 수 있도록 해야 한다.

오류 안내가 없으면 사용자는 어떤 부분에서 문제가 발생했는지 알기 어려워, 이를 수정하지 못해 회원가입 절차를 완료하지 못할 가능성이 높아진다.

입력 오류 발생 시, 오류가 발생한 입력란으로 초점이 이동되지 않아 화면낭독프로그램 사용자나 운동 장애 사용자가 오류 위치를 찾기 어려운 사례이다.

#### 4)	오류 발생 시 입력했던 값 초기화
회원 가입 입력 내용을 제출했을 때 오류가 발생하여, 입력한 모든 값이 초기화되는 사례이다. 이 경우, 사용자는 이전에 입력했던 데이터를 다시 입력해야 하여 불필요한 시간 소모와 작업 반복이 발생한다.

특히, 키보드만 사용하는 사용자에게는 입력란을 다시 탐색하고 값을 입력하는 과정에서 추가적인 작업이 요구되어 작업 시간이 증가할 수 있다. 

이러한 경우는 입력 과정에서 동일한 오류를 반복하게 만들며, 결과적으로 사용자가 회원가입을 완료하기 어렵게 만든다.

# 4. 견고성

견고성이란 콘텐츠가 기술 표준을 준수하여 제작함으로써, 다양한 기술 환경 및 보조 기술과의 호환성을 확보하는 것을 의미한다. 마크업 시 태그를 제대로 열고 닫지 않거나, id 속성이 중복되면 브라우저나 화면낭독프로그램 등에서 콘텐츠를 의도대로 해석하지 못할 수 있다. 또한, 다양한 브라우저와 기기(스마트폰, 태블릿 등)에서도 콘텐츠가 문제없이 접근 가능하도록 고려해야 한다.

다음과 같은 상황에서 도움이 된다.
- 보조 기술(화면낭독프로그램, 음성 입력 시스템 등)과의 연동이 안정적으로 이루어지면, 장애인을 포함한 다양한 사용자가 문제없이 이용할 수 있다.
- 웹 애플리케이션을 오류 없이 표준 API를 활용해 제작하면, 확장이나 외부 연동이 쉬워지고 예기치 못한 호환성 문제를 줄일 수 있다.
- 태그 중첩 오류를 제거하고 시맨틱(semantic) HTML을 준수하면, 새로운 브라우저·기기·운영체제에서도 레이아웃이 깨지지 않고 유지보수가 용이해진다.

견고성이 보장된 콘텐츠는 접근성을 높이며, 기업이나 프로젝트 차원에서도 안정적인 서비스 제공이 가능하다.

## 4.1 문법 준수

<Aside title="검사 항목">
마크업 언어의 요소는 열고 닫음, 중첩 관계 및 속성 선언에 오류가 없어야 한다.
</Aside>

### 목적
마크업 오류 방지 검사항목은 브라우저와 보조 기술이 웹 콘텐츠를 올바르게 해석하도록 보장하기 위한 것이다. 기본 마크업 문법(열고 닫힘, 올바른 중첩, 속성 중복 방지)을 준수하면 모든 사용자가 동일한 정보를 정확하게 전달받을 수 있으며, 접근성 및 호환성이 향상된다.

##	배경 및 필요성
HTML 코드에 문법적인 오류가 있거나 표준을 따르지 않은 요소가 포함되어 있다면, 보조 기술이나 브라우저가 이를 해석하는 방식이 달라질 수 있다.

브라우저는 오류를 자동 보완하려 하지만 보완 방식이 브라우저마다 다르고, 보조 기술은 오류를 제대로 처리하지 못할 수 있다. 이에 따라 사용자에게 제공되는 정보가 왜곡되거나 빠질 가능성이 있다. 따라서 콘텐츠를 개발할 때는 반드시 마크업 언어의 표준을 준수하고, 문법 오류가 발생하지 않도록 주의해야 한다.

이를 위해 웹 사이트 개발 과정에서 [W3C Markup Validator](https://validator.w3.org/)를 활용하여 HTML 문법 오류가 없는지 검사하거나, 개발 도구 또는 개발 과정에 구문 검사 도구를 통합하여 정기적으로 확인하는 것이 좋다.

##	기본 규칙
- 여는 태그와 닫는 태그는 반드시 짝을 이루어 작성하며, 태그명이 일치해야 한다.
- 중첩된 요소는 먼저 열린 요소가 나중에 닫혀야 한다. 예를 들어, span 요소 내에 em 요소가 있으면 닫을 때는 먼저 em 요소를 닫고 그 다음 span 요소를 닫는다.
- 동일한 요소에 중복된 속성이 없도록 하며, 특히 id 속성값은 동일 페이지 내에서 유일해야 한다.

### 적용 사례
#### 1)	열고 닫는 요소의 정확한 구현
HTML 요소는 `<h3>제목</h3>`처럼 여는 태그와 닫는 태그가 올바르게 작성되어야 하며, 이를 통해 브라우저가 요소의 범위를 정확히 인식할 수 있어야 한다.

여는 태그와 닫는 태그가 일치하지 않거나, 중첩된 요소가 올바른 순서로 닫히지 않으면 브라우저와 보조 기술이 이를 다르게 해석하여 콘텐츠의 의미나 구조가 왜곡될 수 있다.

이러한 오류는 키보드 초점 이동에도 영향을 미칠 수 있다. 특히 a 요소와 같은 상호작용 요소에서 닫는 태그가 누락되면, 초점이 의도치 않은 영역으로 확장되거나, 특정 요소에 초점이 정상적으로 접근하지 못하는 문제가 발생할 수 있다. 

##### a)	요소의 열고 닫는 요소 일치
여는 태그가 있다면 닫는 태그도 함께 작성해야 하며, 태그명은 일치해야 한다.
```html
<h2>마크업 오류 방지</h2>
<h3>열고 닫는 태그의 정확한 구현</h3>
<p>여는 태그와 닫는 태그는 각각 &lt;tag&gt;와 &lt;/tag&gt;로 구현되어야 한다.</p>
```

##### b)	요소 중첩 시 올바른 열기와 닫기 순서 구현
HTML 요소는 중첩 구조를 가지므로, 요소를 열 때와 닫을 때 순서를 반드시 지켜야 한다. 먼저 열린 요소는 나중에 닫히는 방식으로 작성해야 한다.

만약 순서를 지키지 않는다면, 화면낭독프로그램이나 일부 브라우저는 요소의 기능을 올바르게 해석하지 못할 수 있다. 이에 따라 강조 표현이 의도와 다르게 전달되거나, 중첩된 요소가 모두 하나의 스타일로 처리되는 문제가 발생할 수 있다.
```html
<span>
    &lt;span&gt; 태그를 열고 이어서 &lt;em&gt; 태그를 열었다면, 닫을 때는 
    <em>
        &lt;em&gt; 태그를 먼저 닫고,
    </em>
    그다음 &lt;span&gt; 태그를 닫아야 한다.
</span>
```
#### 2)	동일한 속성 중복으로 사용 방지
하나의 HTML 요소에 중복된 속성을 지정하지 않도록 해야 한다. 중복된 속성을 사용하면 브라우저가 어떤 값을 적용해야 할지 명확하지 않아, 마지막으로 선언된 값을 적용하거나, 브라우저마다 다르게 처리될 수 있다. 이에 따라 웹 페이지의 동작이 예기치 않게 변경되거나, 보조 기술에서 오류가 발생할 가능성이 있다.

##### a)	중복된 id 방지
페이지 내에서 동일한 id 값이 중복되지 않도록 작성해야 한다.

id가 중복되면, label 요소 연결(예: for 속성)이나 ARIA 속성(예: aria-labelledby, aria-describedby)에서 참조가 올바르게 이루어지지 않아 보조 기술 사용자가 잘못된 정보를 전달받을 수 있다. 또한, CSS 스타일 적용이 의도와 다르게 이루어지거나, JavaScript가 특정 요소를 정확히 참조하지 못하는 문제를 발생시킬 수 있다.

이러한 문제를 방지하기 위해 id는 페이지 내에서 고유하게 설정해야 한다.

```html
<label for="user-id">아이디</label>
<input type="text" id="user-id">
<label for="user-pwd">비밀번호</label>
<input type="password" id="user-pwd">
```

##### b)	속성 중복 사용 방지
하나의 요소에서 동일한 속성을 중복 사용하지 않아야 한다.

동일한 속성이 중복될 경우, 브라우저는 HTML을 해석 과정에서 우선순위를 명확히 판단하지 못해 중복된 값 중 하나만 선택해 적용할 수 있다. 

이에 따라 CSS의 스타일이 의도와 다르게 적용되거나, JavaScript가 특정 속성을 참조할 때 잘못된 값을 가져와 기능이 오작동할 가능성이 있다. 따라서 코드 작성 시 속성 중복 여부를 점검하여 이러한 문제를 방지해야 한다.

```html
<p style="color:red" style="color: blue">공지사항</p>
```

### 개선 필요 사례
#### 1)	닫는 요소 누락
다음은 h3 요소의 닫는 태그가 누락되어 이후 내용까지 제목으로 인식하는 사례이다.

Chrome 브라우저는 h3 요소가 닫히지 않았다고 판단하여, 뒤따르는 p 요소와 그 안의 모든 내용을 h3 요소의 일부로 포함한다. 그 결과, '인생의 의미에 대한 탐구'뿐만 아니라 본문 내용까지 모두 제목으로 인식한다. 

```html
<h3>인생의 의미에 대한 탐구
<p>
    인생은 우리가 매일 경험하는 다양한 사건과 감정으로 가득 차 있습니다.
    이러한 경험들은 우리의 성격을 형성하고,
    우리가 세상과 어떻게 상호작용하는지를 결정짓는 중요한 요소입니다.
</p>
```

```html
<h3>인생의 의미에 대한 탐구</h3>

    이러한 경험들은 우리의 성격을 형성하고,
    우리가 세상과 어떻게 상호작용하는지를 결정짓는 중요한 요소입니다.
</p>
```

#### 2)	잘못된 중첩 순서
다음은 중첩된 요소의 열기와 닫기 순서가 잘못된 사례이다. 요소를 열고 닫는 순서가 맞지 않아, 브라우저가 이를 자동으로 수정하여 렌더링하면서 span 요소와 em 요소의 구조가 원래 의도와 다르게 변경되었다.

이로 인해 강조된 텍스트의 범위가 달라지고, 화면낭독프로그램이 이를 정확히 인식하지 못하게 된다. 결과적으로 보조 기술은 강조된 부분을 혼동하거나 중요한 정보를 놓칠 위험이 생긴다.

```html
<span>
	&lt;span&gt; 태그를 열고 이어서 &lt;em&gt; 태그를 열었다면, 닫을 때는
	<em>
		&lt;em&gt; 태그를 먼저 닫고,
	</span>
    그다음 &lt;span&gt; 태그를 닫아야 한다.
</em>
```

##### 개선방안
중첩된 요소의 열고 닫는 순서를 올바르게 작성해야 한다.

`span` 요소 안에 `em` 요소를 중첩하려면 `span` 요소를 먼저 열고 `em` 요소를 연 뒤, 닫을 때는 반대 순서로 em 요소를 먼저 닫고 그다음에 span 요소를 닫아야 한다.

#### 3)	한 페이지 내 중복된 id 사용
두 개의 input 요소에 모두 ‘user-id’라는 동일한 id 값이 지정된 사례이다.

이에 따라 화면낭독프로그램이 label 요소의 for 속성을 통해 입력란을 정확히 연결하지 못하고, ‘아이디’와 ‘비밀번호’가 어떤 입력란에 연결되는지 사용자에게 제대로 된 정보를 제공하지 못한다. 

```html
<label for="user-id">아이디</label>
<input type="text" name="user-id" id="user-id">
<label for="user-pwd">비밀번호</label>
<input type="password" name="user-pwd" id="user-id">
<button type="submit">로그인</button>
```

##### 개선방안
각 input 요소에 고유한 id 값을 부여하고, label 요소의 for 속성도 이에 맞게 수정해야 한다.

```html
<label for="user-id">아이디</label>
<input type="text" name="user-id" id="user-id">
<label for="user-pwd">비밀번호</label>
<input type="password" name="user-pwd" id="user-pwd">
<button type="submit">로그인</button>
```

#### 4)	속성 중복 사용
한 요소 내 `style` 속성이 중복하여 사용된 사례이다.

브라우저는 중복된 `style` 속성을 처리할 때, 특정 값만 적용하거나 브라우저별로 다르게 동작할 수 있다. 이로 인해 사용자가 의도한 스타일이 일관되게 적용되지 않는 문제가 발생할 수 있다.

마지막 선언 된 값이 적용되는 경우, `display:none`의 영향으로 해당 콘텐츠를 보조 기술이 인식하지 못하게 된다.

```html
<a href="..." style="opacity:0" style="display:none;" ...>... </a>
```

##### 개선방안
동일한 요소 내에서 `style` 속성을 중복 선언하지 않도록 한다.

## 4.2 웹 애플리케이션 접근성

<Aside title="검사 항목">
콘텐츠에 포함된 웹 애플리케이션은 접근성이 있어야 한다.
</Aside>

### 목적
웹 콘텐츠에 포함된 플러그인이나 애플리케이션이 보조 기술을 사용하는 사용자도 문제없이 접근하고 사용할 수 있도록 보장하기 위함이다.

##	배경 및 필요성
웹 애플리케이션과 플러그인은 단순한 정보 제공을 넘어 사용자와의 상호작용, 데이터 처리, 동적 콘텐츠 제공 등 다양한 기능을 수행한다. 

이러한 기능이 보조 기술과 호환되지 않거나 키보드 조작을 지원하지 않으면, 키보드 사용자와 보조 기술 사용자는 콘텐츠를 인식하거나 조작할 수 없는 문제가 발생할 수 있다. 또한, 시각적 정보만으로 콘텐츠를 이해하기 어려운 색각 이상 사용자나, 특정 환경에서 플러그인이나 기술이 특정 운영 체제 또는 보조 기술과 호환되지 않는 경우 접근성이 크게 저하될 수 있다. 이로 인해 일부 사용자는 정보에 접근할 수 없거나, 중요한 기능을 사용할 수 없는 문제가 발생한다.

따라서 웹 콘텐츠에 포함된 플러그인과 웹 애플리케이션은 보조 기술 호환성을 고려하고, 키보드 내비게이션을 지원하며, 시각적 정보 외에도 다양한 대체 수단을 제공하는 등 앞서 다룬 모든 검사항목에 대한 접근성이 보장되어야 한다.

##	기본 규칙
- 플러그인과 웹 애플리케이션은 화면낭독프로그램, 화면 확대 프로그램 등 주요 보조 기술과 호환되어야 한다.
- 마우스 사용이 어려운 사용자도 키보드만으로 모든 기능에 접근하고 조작할 수 있도록 인터페이스가 설계되어야 한다.
- 다양한 브라우저, 운영 체제, 보조 기술 환경에서 웹 애플리케이션과 플러그인의 접근성을 테스트하고 개선해야 한다.
- 플러그인이나 웹 애플리케이션을 선택할 때는 접근성 지원 여부를 검토하고, 접근성이 확보된 솔루션을 채택해야 한다.

### 용어
<dl>
	<dt>플러그인</dt>
	<dd>웹 브라우저에서 기본적으로 지원하지 않는 특정 기능이나 콘텐츠를 제공하기 위해 추가 설치하는 소프트웨어이다. 플러그인은 웹 페이지와 통합되어 사용자가 특정 콘텐츠를 원활하게 이용할 수 있도록 돕는 역할을 한다.</dd>

### 적용 사례
#### 1)	접근성을 준수하는 부가 웹 애플리케이션 제공
부가 웹 애플리케이션을 사용하는 경우, 사전에 접근성 지침 준수 여부(대체 텍스트 제공, 키보드 접근성, 화면낭독프로그램 호환성 등)를 충분히 검토하는 것이 필수적이다. 부가 웹 애플리케이션은 직접 개발하거나 수정할 수 없는 경우가 많으므로, 접근성 지원 사항을 미리 확인하는 것이 중요하다.

숙소 예약 사이트에서 지도 서비스를 부가 웹 애플리케이션으로 사용할 때, 해당 서비스가 키보드 탐색이 가능한지, 화면낭독프로그램을 통해 상호작용 요소의 이름과 기능이 명확히 전달되는지 등을 점검해야 한다.

사용한 부가 애플리케이션이 접근성을 완전히 충족하지 못한다면, 추가 설명이나 대체 수단을 제공하여 모든 사용자가 동등하게 콘텐츠를 이용할 수 있도록 개선해야 한다.

#### 2)	리치 웹 애플리케이션(RIA)의 접근성 보장
사용자 정의 컴포넌트와 같은 리치 웹 애플리케이션 요소는 기본 HTML 요소가 제공하는 접근성 정보를 상속받지 못하는 경우가 많다. 또한, 탭(Tab) UI와 같이 HTML 요소만으로는 역할 및 상태 정보를 충분히 전달하기 어려운 동적 콘텐츠의 경우가 있으므로, ARIA 속성을 적절히 활용하여 보조 기술에서 해당 컴포넌트의 역할과 상태를 해석할 수 있도록 제공할 수 있다.

<figure>
	![](/images/image269.png)
	<figcaption>이미지. ARIA Authoring Practices Guide (APG), Select-Only Combobox Example 사례 (출처: W3C)</figcaption>
</figure>

예를 들어, 사용자 정의 콤보박스를 구현할 때, 다음과 같이 적용할 수 있다.
- 메뉴 전체에 `role="listbox"`와 각 옵션에 `role="option"`을 사용하고,
- 메뉴의 열림/닫힘 상태는 `aria-expanded="true/false"`로,
- 선택된 항목은 `aria-selected="true"`설정한다.

또한, 콤보박스는 기본 HTML 콤보박스와 동일한 키보드 조작(예: 옵션 목록 펼침은 Enter/Space, 닫힘 및 선택 취소는 Esc, 옵션 탐색은 상하 화살표)이 가능하도록 제공해야 한다.

이와 같이 ARIA 속성을 활용하면, 시각장애인이나 키보드 전용 사용자가 사용자 정의 컴포넌트를 보다 쉽게 인식하고 조작할 수 있다. 다만, ARIA 속성은 기본 HTML 요소의 기능이나 조작을 대체하지 않으므로 자바스크립트 등 구현이 추가로 필요하여, 가능한 경우 HTML 기본 기능을 우선으로 활용하는 것이 바람직하다.

<Card title="참고. ARIA Authoring Practices Guide" icon="open-book">
사용자 정의 컴포넌트를 구현할 때는 W3C의 [ARIA Authoring Practices Guide](https://www.w3.org/WAI/ARIA/apg/)를 참고하면, 다양한 컴포넌트에 대한 접근성 패턴, ARIA 속성 사용 방법, 그리고 키보드 인터페이스 설계 원칙을 상세히 제공하여 표준에 기반한 개발에 도움을 받을 수 있다.

### 개선 필요 사례
#### 1)	접근성이 미흡한 웹 애플리케이션 제공
다음은 도서관 사이트에서 뷰어 시스템이 적용되었으나 키보드로 페이지를 제어할 수 없는 사례이다.

웹 뷰어를 마우스로 사용 시에는 정상적으로 목록, 페이지 보기, 설정, 검색, 이전/다음 버튼 등 다양한 기능을 사용할 수 있지만, 키보드 탐색 시에는 해당 기능 버튼이 화면에 노출되지 않아 사용할 수 없다. 이는 키보드 사용자를 고려하지 않은 설계로 인해 웹 애플리케이션의 접근성이 저하된 사례이다.

또한, 화면낭독프로그램 사용자가 라디오 버튼 옵션을 키보드 방향키로 제어할 때, 옵션 변경과 동시에 책 페이지가 변경되어 의도치 않은 페이지 이동이 발생하는 문제가 있다. 이러한 문제는 키보드 상호작용 설계가 적절하지 않아 사용자가 원하는 정보를 인식하거나 조작할 수 없게 만든다.

<figure>
	![](/images/image271.png)
	<figcaption>이미지. 웹 뷰어에서 마우스 호버 시 노출되는 메뉴 버튼(위쪽)과 키보드 접근 시에는 노출되지 않는 화면 비교 사례</figcaption>
</figure>

##### 개선 방법
외부 프로그램을 도입하기 전에 해당 프로그램이 키보드 내비게이션, 보조 기술 호환성, 그리고 시각적·청각적 정보의 대체 수단 제공 등 접근성 요구사항을 충족하는지 확인해야 한다.

만약 접근성 요구사항을 충족하지 못하는 프로그램이라면, 동일한 기능을 제공하면서 접근성을 보장할 수 있는 다른 대안을 검토해야 한다. 이러한 기준을 충족하는 프로그램을 선택함으로써, 다양한 사용자 그룹이 콘텐츠를 동등하게 이용할 수 있는 환경을 마련할 수 있다.

#### 2)	접근성 일부만 준수한 웹 애플리케이션
접근성이 일부만 준수되어 키보드 및 보조 기술 사용자에게 불편을 주는 공동인증서 로그인 웹 애플리케이션 사례이다.

배너에 대체 텍스트가 제공되지 않아, 화면낭독프로그램 사용자는 해당 정보를 그래픽 링크로만 인식하게 되어 배너의 목적과 내용을 파악하기 어렵다. 탭 키를 사용하여 탐색할 때 인증서 암호 입력은 가능하지만, 인증서 개수가 많을 경우 원하는 인증서를 선택하기 어렵다.

또한, 순차 탐색 시 화면에 표시되는 콘텐츠 순서와 실제 탐색 순서가 일치하지 않아, 사용자가 논리적인 흐름대로 탐색할 수 없다. 인증서 목록은 표 형태로 제공되지만, 각 항목이 개별 요소로 분리되어 있어 정보의 연관성을 파악하기 어렵다.

이에 따라 보조 기술을 사용하는 사용자는 인증 과정에서 불편함을 겪게 된다.

<figure>
	![](/images/image272.png)
	<figcaption>이미지. 공동인증서 로그인 시 실행된 웹 애플리케이션 사례</figcaption>
</figure>

##### 개선 방법
웹 애플리케이션 또한 앞에서 다룬 검사 항목을 모두 준수하는지 점검해야 한다.

이미지 배너에 대체 텍스트를 제공하고, 키보드 사용을 보장하고, 초점은 논리적으로 이동하고, 표는 보조 기술 사용자가 행과 열의 관계를 쉽게 이해할 수 있도록 설계하고, 탐색이 쉽도록 개선한다.

이러한 개선을 통해, 키보드 및 보조 기술 사용자도 인증 절차를 원활하게 수행할 수 있도록 해야 한다.

#### 3)	접근성이 준수되지 않은 사용자 정의 콤보박스
다음은 상품 상세 페이지의 옵션 선택 영역이 콤보박스로 인식되지 않는 사례이다.

<figure>
	![](/images/image273.png)
	<figcaption>이미지. 커스텀된 콤보박스 사례</figcaption>
</figure>

옵션 목록이 `ul` 요소로 구현되어 있어 보조 기술 사용자는 이를 선택 가능한 콤보박스가 아닌 단순한 목록으로 인식할 수 있다. `a` 요소로 제공된 각 옵션은 페이지 이동 링크로 인식되어, 사용자가 옵션 목록을 제대로 탐색하거나 상호작용을 하기 어렵다.

또한, 옵션 목록의 열림/닫힘 상태 및 선택 상태가 명확하게 제공되지 않아, 사용자가 자신이 선택한 옵션을 확인하거나 변경하는 데 어려움을 겪는다.

```html
<div class="prd_option_box" id="buy_option_box">
	<a href="javascript:;" class="sel_option" id="buyOpt">
		상품을 선택해주세요
	</a>
	</div>
	<ul class="sel_option_list" id="option_list">
	<li class="type1">
		<a style="cursor:pointer" onclick="javascript:...">
			<span class="color"> <img src="..." alt=""> </span>
			<div class="set">
				<span class="option_value">
					[1+1 한정] 핸드크림
					<span class="option_price"> 7,900원 </span>
				</span>
			</div>
		</a>
	</li>
	...
</ul>
```

##### 개선 방법
콤보박스는 `select` 요소를 사용하여 구현하는 것이 가장 적합하다. 이는 기본적으로 키보드 탐색과 역할 안내, 선택 상태를 제공하므로 추가 작업이 필요 없다.

만약 사용자 정의 컴포넌트를 사용해야 한다면, `role="listbox"`와 `role="option"`을 사용해 보조 기술이 이를 선택 가능한 옵션으로 인식하도록 하고, 열림/닫힘 상태 정보를 `aria-expanded` 등의 ARIA 속성을 통해 제공해야 한다. 또한, 키보드로 탐색할 수 있도록 방향키, `Enter` 키, `Space` 키 등의 적절한 키보드 탐색을 구현해야 한다.

<Card title="참고. WAI-ARIA를 사용하기 전에 알아야 할 것들" icon="open-book">
WAI-ARIA는 웹 접근성을 높이기 위한 강력한 도구이지만, 올바르게 사용하지 않으면 오히려 접근성을 저하시킬 수 있다. 따라서 다음 사항을 반드시 확인하고 활용해야 한다.

- **나쁜 ARIA 보다 ARIA가 없는 것이 낫다.**<br/>
기본 HTML만으로 접근성을 충분히 제공할 수 있다면 WAI-ARIA를 사용하지 않는다. 예를 들어, 버튼은 `<div role="button">`이 아닌, 키보드 접근성과 클릭 이벤트를 제공하는 button 요소를 사용한다.
- **WAI-ARIA는 보조기기를 위한 정보 제공 도구임을 이해한다.**<br/>
aria-*와 role 속성은 시각적 스타일이나 동작을 변경하지 않으며, 보조 기술과의 소통을 돕는 역할만 한다. `<div role="button">`은 보조 기술에서 버튼으로 인식되지만, 키보드 접근성은 제공하지 않는다. 이 때, 초점을 받을 수 있도록 하려면 tabindex="0"을 추가해야 한다. 
- **aria-hidden은 신중하게 사용해야 한다.**<br/>
aria-hidden="true"는 보조 기술이 해당 요소를 무시하도록 한다. 따라서 링크, 버튼, 입력란과 같이 상호작용이 필요한 요소에는 사용하지 않는다. 숨겨진 요소가 다시 표시될 경우 aria-hidden=”false”로 변경해야 한다.
만약, 숨겨진 콘텐츠가 초점을 받지 않도록 해야 한다면 tabindex="-1"을 함께 적용해야 한다.
- **실제 환경에서 반드시 테스트한다.**<br/>
WAI-ARIA를 적용한 콘텐츠는 실제 보조 기술 환경에서 반드시 테스트해야 한다. 특정 브라우저나 보조 기기에서 예상과 다르게 동작할 수 있으며, 지원하지 않는 기능이나 버그가 존재할 수 있다.

자세한 내용은 앞서 소개한 W3C의 [ARIA Authoring Practices Guide](https://www.w3.org/WAI/ARIA/apg/)를 참조한다.

- [Using ARIA](https://www.w3.org/TR/using-aria/)

# 5. KWCAG 이해와 실무 적용

한국형 웹 콘텐츠 접근성 지침 2.2(이하 KWCAG 2.2)는 다양한 사용자가 웹 콘텐츠를 동등하게 인식하고 활용할 수 있도록 보장하기 위해 마련된 지침이다. 2022년 12월, 기존 KWCAG 2.1을 개정하면서 국제 표준(WCAG 2.1)의 내용을 반영하여 KWCAG 2.2가 제정되었다.

본 문서는 웹 제작자가 KWCAG 2.2의 접근성 검사 항목을 보다 쉽게 이해하고 실무에 적용할 수 있도록 돕고자 한다.

### KWCAG와 WCAG의 관계
#### WCAG의 흐름
웹 접근성 표준은 W3C의 WCAG(Web Content Accessibility Guidelines)을 중심으로 발전해 왔으며, 국제적으로 가장 널리 적용되는 지침이다. W3C는 기술의 발전과 다양한 사용자의 요구를 반영하여 지속적으로 접근성 지침을 개정하고 있다.

2018년에 발표된 WCAG 2.1은 모바일 기기의 확산과 저시력, 인지 장애, 학습 장애가 있는 사용자의 요구를 적극적으로 반영했다. 특히 음성 인식을 사용하는 사용자를 위한 '문자 단축키', '레이블과 네임', 터치 기반 환경을 고려한 '포인터 제스처', '포인터 입력 취소' 그리고 기기 움직임 등 신체 동작을 입력 수단으로 사용할 때 고려해야 하는 '동작기반 작동'과 같은 새로운 항목들이 추가되었다. 

다음은 WCAG 2.1에서 중요도 A(Level A) 중심으로 추가된 항목 목록이다. 
- 2.1.4 Character Key Shortcuts (Level A, 문자 단축키)
- 2.5.1 Pointer Gestures (Level A, 포인터 제스처)
- 2.5.2 Pointer Cancellation (Level A, 포인터 입력 취소)
- 2.5.3 Label in Name (Level A, 레이블과 네임)
- 2.5.4 Motion Actuation (Level A, 동작기반 작동)

2024년 표준으로 발표된 WCAG 2.2에서는 사용자 지원을 위한 '찾기 쉬운 도움 정보', 반복적인 데이터 입력을 최소화하는 '반복 입력 정보', 인지 능력에만 의존하지 않는 인증 방식을 요구하는 '접근 가능한 인증' 항목이 추가되어 더욱 폭넓은 사용자 경험 개선이 이루어졌다.

그 중 KWCAG 2.2에서 참고한 항목은 다음과 같다.
- 3.2.6 Consistent Help (Level A, 찾기 쉬운 도움 정보)
- 3.3.7 Redundant Entry (Level A, 반복 입력 정보)
- 3.3.8 Accessible Authentication (Minimum) (Level AA, 접근 가능한 인증)

이 중 WCAG 2.2 초안에서 논의되었던 '고정된 참조 위치 정보(Fixed Reference Points)' 성공 기준은 기술적 복잡성과 실무 적용성 문제로 충분한 합의에 이르지 못해 최종 발표 단계에서 삭제되었다.

한편, 초안이 발표된 차세대 웹 접근성 표준인 [WCAG 3.0](https://www.w3.org/TR/wcag-3.0/)에서는 보다 유연하고 포괄적인 접근성 평가 방식을 제시할 예정이며, 다양한 사용자 환경과 기술 변화에 효과적으로 대응할 수 있도록 설계되고 있다.

#### KWCAG 2.2와 2.1차이
한국형 웹 콘텐츠 접근성 지침(KWCAG)은 국제 표준 WCAG의 Level A 항목을 중심으로 국내 환경에 적합하게 재구성한 지침이다.

KWCAG 2.1은 2015년 3월 개정된 이후 약 8년 동안 큰 변화가 없었다. 그동안 웹 환경과 기술이 급속히 발전했으며, 모바일 기기의 보급 확대와 함께 다양한 사용자 환경이 등장하면서 새로운 접근성 요구사항들이 발생했다. 특히, 터치스크린을 사용하는 모바일 환경에서의 사용성 개선, 키보드 이외의 입력방식 지원 확대, 그리고 인지장애를 가진 사용자를 위한 고려가 더욱 중요해졌다.

이에 따라 2022년 12월, 최신 국제표준 WCAG 2.2를 반영한 한국형 웹 콘텐츠 접근성 지침 2.2(KWCAG 2.2)가 새롭게 제정되었다. KWCAG 2.2는 기존의 4대 원칙 체계를 유지하면서, 총 14개 지침과 33개의 검사항목으로 구성되었다.

KWCAG 2.1과 비교하면 '적응성'과 '입력 방식' 지침이 추가되었으며, '콘텐츠의 논리성' 지침이 삭제되어 총 14개 지침으로 개편되었다.

또한 사용 환경 변화와 국제적 흐름에 따라 검사항목에서도 '문자 단축키', '고정된 참조 위치 정보', '단일 포인터 입력 지원', '포인터 입력 취소', '레이블과 네임', '동작기반 작동', '찾기 쉬운 도움 정보', '접근 가능한 인증', '반복 입력 정보'의 총 9개 항목이 추가되었다. 

이러한 개정을 통해 더 많은 사용자가 웹 콘텐츠에 쉽게 접근하고 이용할 수 있도록 하는데 초점을 맞췄다.

<Card title="참고. 중요도(Level A, AA, AAA)" icon="open-book">
WCAG는 성공 기준(Success Criterion, SC)의 웹 접근성 준수 수준을 Level A, AA, AAA 세 단계로 구분하며, 이는 사용자에게 미치는 영향도를 반영한다.

각 단계의 특징은 다음과 같다.
- Level A: 기본적인 접근성 기준으로, 장애가 있는 사용자가 콘텐츠를 이용할 수 있도록 보장하는 최소한의 필수 요건이다. 이를 준수하지 않으면 시각 및 청각장애 사용자는 콘텐츠를 전혀 이용할 수 없을 수도 있다. 
- Level AA: 해당 레벨을 충족하면 대다수의 장애 사용자가 콘텐츠에 접근할 수 있음을 말한다. WCAG 준수를 요구하는 많은 국가와 기관에서 Level AA 준수를 기본 요건으로 삼고 있으며, 법적 요구 사항으로 지정되는 경우가 많다.
- Level AAA: 거의 모든 사용자가 콘텐츠에 접근할 수 있도록 하는 높은 수준의 접근성을 보장한다. 그러나 모든 콘텐츠에 적용하기 어려울 수 있으며, 특정 사용자층을 위한 맞춤형 접근성이 필요할 때 권장된다.
</Card>

### 웹 접근성 필수 구성 요소
웹 접근성은 다양한 구성 요소가 상호 연관되어 작동한다. 웹 제작자는 웹 접근성 지침을 올바르게 이해하고 준수하기 위해 각 구성 요소의 역할과 상호작용 방식을 이해하는 것이 필요하다.
- 콘텐츠 제작: 저작 도구를 사용하여 웹 접근성 표준에 맞는 콘텐츠를 제작하고, 대체 텍스트와 같은 필수 접근성 기능을 포함해야 한다.
- 저작 도구: 접근성을 지원하는 웹 콘텐츠 제작 도구로, 콘텐츠 제작자가 접근성을 준수한 콘텐츠를 보다 쉽게 제작할 수 있도록 돕는다.
- 브라우저 및 보조 기술: 웹 콘텐츠를 사용자에게 전달하는 플랫폼이며, 장애인이 콘텐츠를 이해하고 상호작용할 수 있도록 화면낭독프로그램, 화면 확대기 등 보조 기술과 함께 작동한다.
- 평가 도구: 웹 콘텐츠가 접근성 기준을 충족하는지 자동 또는 수동으로 평가하고, 발견된 문제를 확인하여 개선할 수 있도록 돕는다.

이러한 요소들이 협력하여 장애 유무와 관계없이 누구나 웹 콘텐츠를 원활하게 이용할 수 있도록 지원한다.

### WCAG의 문서 구조
WCAG의 문서 구조와 내용을 이해하면 KWCAG를 이해하는 데 도움이 된다. 다음 문서들은 웹 접근성 지침을 이해하고 실무에서 효과적으로 활용하는 데 유용한 자료들이다.

###	[WCAG 2.2](https://www.w3.org/TR/WCAG22/) (표준)

WCAG 2.2는 웹 콘텐츠의 접근성을 높이기 위한 국제 표준으로, 웹 접근성 준수를 위한 핵심 문서이다. 이 문서는 원칙(Principles), 지침(Guidelines), 성공 기준(Success Criteria)으로 구성되어 있다.

- 웹 접근성의 기본 기준을 명확히 이해하고 정의할 때
- 국제 프로젝트에서 웹 접근성 요건을 설정할 때
- 웹 접근성 정책을 수립할 때

###	빠른 참조, [How to Meet WCAG](https://www.w3.org/WAI/WCAG22/quickref/)

W3C에서 제공하는 웹 콘텐츠 접근성 지침(WCAG)의 요구사항과 기술을 빠르게 참조할 수 있는 맞춤형 도구이다.

- WCAG 2의 모든 지침과 성공 기준을 한눈에 확인 가능
- WCAG 2 체크리스트 제공
- 맞춤형 설정 지원 (Level A, AA, AAA)
- HTML, CSS, ARIA 등 기술별 필터링 기능 제공

###	해설서, [WCAG Understanding](https://www.w3.org/WAI/WCAG22/Understanding/)

WCAG 지침과 성공 기준의 목적, 배경, 사용자에게 미치는 영향을 상세히 설명하는 문서이다. 지침이 왜 필요한지 이해하는 데 도움을 주며, 관련된 기술 문서(Techniques) 링크도 포함되어 있다.

- 성공 기준(Success Criterion, SC)의 의도 설명
- 장애를 가진 사용자에게 미치는 영향
- 브라우저 및 보조 기술 관련 설명
- 예시 및 참고자료 제공

###	제작 기법, [WCAG Techniques](https://www.w3.org/WAI/WCAG22/Techniques/)

웹 콘텐츠에서 WCAG 지침을 실제로 구현하는 방법을 설명하는 문서로, 개발자가 접근성 준수를 위한 구체적인 코딩 기법을 학습할 수 있다.

- HTML, CSS, Javascript, WAI-ARIA 등의 접근성 구현 기법 제공
- 멀티미디어 및 동적 콘텐츠 접근성 적용 방법 포함
- WCAG 성공 기준과 연계된 기술적 해결책 제시

###	접근성 적합성 점검 규칙, [ACT Rules](https://www.w3.org/WAI/standards-guidelines/act/rules/)

웹 접근성 평가 및 테스트를 위한 구체적인 규칙을 정리한 문서로, WCAG 성공 기준의 적합성을 검증하는 방법을 설명한다. 

- WCAG 2 ACT 규칙 소개 및 활용 방법
- 웹 접근성 평가 도구 개발 및 테스트 기준 제공
- WCAG 2.2 성공 기준에 대한 평가 지침 포함

## 실무 적용

### 단계별 웹 접근성 실무 적용 가이드

웹 접근성의 준수는 기획, 디자인, 개발 등 모든 역할에서 긴밀히 협력해야 이루어질 수 있다. 각 역할에 맞는 웹 접근성 실무 적용 방안의 예시는 다음과 같다.

- 기획
	- 사용자 요구사항에 접근성 기준 포함
	- 콘텐츠 및 상호작용 방식을 접근성을 고려하여 기획
- 디자인
	- 명도 대비, 색상 사용, 텍스트 크기, 버튼 크기 등 접근성을 고려한 UI 디자인
	- 사용자 인터페이스의 일관성 유지
- 개발
	- HTML, CSS, ARIA 등 기술 표준에 맞춰 개발
	- 접근성 준수 코드 작성 및 테스트
- 점검
	- 접근성 평가 도구 활용하여 콘텐츠 검증
	- 접근성 준수 여부를 지속적으로 모니터링 및 관리

이러한 접근성 적용 방안을 기반으로, 조직에 맞는 최적의 접근성 전략을 수립하는 과정이 필요하다.

### 웹 접근성 업무 프로세스 가이드

본 가이드는 웹 접근성 개선 프로젝트를 처음 수행하는 제작자를 위해 업무를 수행할 수 있도록 기본 프로세스를 다뤄본다.

1. 업무 준비 단계
	- 웹 접근성 관련 기준 확인 (한국형 웹 콘텐츠 접근성 지침 2.2, 웹 접근성 품질 인증 심사 지침)
	- 웹 접근성 표준 및 관련 법률 숙지
	- 접근성 관련 도구 선정 및 설치 (자동 접근성 진단 도구-openWAX, WAVE, axe-core, 화면낭독프로그램- 센스리더, NVDA, VoiceOver, 명도 대비 검사도구- CCA 등)
2. 현황 진단 및 분석
	- 웹사이트 진단 및 평가
	- 자동 점검 도구를 활용한 초기 진단 (예: openWAX, WAVE, axe-core)
	- 키보드만 사용한 수동 진단
	- 화면낭독프로그램(NVDA, VoiceOver, TalkBack 등)을 사용한 수동 진단
	- 접근성 체크리스트 기반 평가
	- 발견된 문제점 목록화 및 우선순위 설정
3. 계획 수립
	- 접근성 개선 일정 및 담당자 설정 (예: 1개월 단위 일정표 양식)
	- 기획, 디자인, 개발자 협업 방안 설정
	- 역할 분담 및 책임자 지정
	- 내부 협업 및 커뮤니케이션 전략 제시 (예: 주간 점검 회의 운영 방안)
4. 디자인/개발 단계
	- 사례 기반 접근성 개선 코드 활용
	- 적절한 대체 텍스트 작성 사례 참고
	- 자막 및 수어를 포함한 멀티미디어 콘텐츠 제작
	- 텍스트 명도 대비 기준 준수 (명도 대비 4.5:1 이상)
	- 키보드만으로 이용 가능하도록 모든 기능 설계
5. 내부 접근성 평가 및 검수
	- 자체 점검 체크리스트를 통한 평가 수행
	- 화면낭독프로그램와 키보드만으로 콘텐츠 이용 가능성 최종 점검
	- 발견된 추가 오류 수정 및 보완
6. 외부 인증 및 사용자 테스트 고려
	- 공식 인증기관에 접근성 품질 인증 신청
	- 사용자 심사 대응을 위한 준비 사항 및 실제 대응 사례 검토
7. 유지 보수
	- 접근성 체크리스트를 활용한 정기 모니터링 계획 (예: 월별, 분기별, 연 1회)
	- 콘텐츠 업데이트 등 접근성 변경 사항 지속 모니터링
	- 접근성 점검 결과 기록 및 보고서 작성 예시 제공
